# 개발 일지: v0.11.x 레퍼런스 soft delete/restore API 도입

**일자:** 2026-04-04
**관련 버전:** v0.11.x
**관련 역할:** spec, build, qa, docs

## 배경 (Context)
기존 Refentra는 레퍼런스를 저장하고 다시 조회하는 흐름은 갖췄지만, 한 번 저장한 데이터를 제품 안에서 정리하는 공식 경로가 없었습니다. 이 상태에서는 잘못 저장한 링크나 중복 저장한 항목을 사용자가 스스로 정리할 수 없어 아카이브 신뢰도가 떨어졌고, 삭제 기능을 도입하더라도 hard delete로 바로 가면 실수 복구가 불가능해지는 문제가 있었습니다.

이번 사이클에서는 `$spec` 단계에서 `docs/prd/refentra-v2.0-reference-delete-restore-lifecycle.md`를 승인했고, 그 결과를 GitHub 이슈 `#15`, `#16`, `#17`로 분해했습니다. 이 일지는 그중 `#15`인 백엔드 삭제/복구 API 구현과 `$qa` 승인까지를 기록합니다.

## 문제 (Problem)
삭제 기능은 단순히 `DELETE` 엔드포인트를 추가한다고 끝나지 않았습니다. 아래 제약을 동시에 만족해야 했습니다.

1. 실수 삭제를 복구 가능하게 만들어야 한다.
2. 다른 사용자의 데이터 존재 여부를 과하게 노출하면 안 된다.
3. 기존 `GET /api/v1/references`의 active-only 조회 계약과 태그 집계가 깨지면 안 된다.
4. 실제 배포 전 단계라 background purge나 전체 휴지통 UI까지 한 번에 넓히면 범위가 과해진다.

## 시도한 것들 (Attempts)
1. **PRD 기준 재정의:** 처음에는 “삭제 기능”만 추가하려고 보면 hard delete가 가장 싸 보였지만, 현재 모델에 이미 `deleted_at`이 있고 조회 SQL도 active 데이터 중심이라 방향이 맞지 않았습니다. 그래서 `$spec`에서 `soft delete + 24시간 restore window + 본인만 삭제/복구`로 정책을 고정했습니다.
2. **기존 create store 확장:** 레퍼런스 핸들러는 이미 fake store 주입으로 테스트 가능한 구조였기 때문에, 새로운 서비스 계층을 만들기보다 기존 store 인터페이스에 `FindReferenceForOwner`, `SoftDeleteReference`, `RestoreReference`를 확장하는 쪽이 변경 비용이 낮았습니다.
3. **QA 경계 테스트 보강:** 초기 구현은 핵심 delete/restore 시나리오는 있었지만, 복구 비인증 `401`과 “삭제되지 않은 항목 복구” `409`가 빠져 있었습니다. `$qa`에서 이 두 경계 케이스를 추가해 상태 코드 정책을 더 단단하게 고정했습니다.

## 최종 해결 (Resolution)
백엔드에 아래 정책을 구현했습니다.

- `DELETE /api/v1/references/:id`
  - 로그인 사용자 기준 본인 소유 항목만 대상
  - 미존재 또는 타인 항목은 `404`
  - 이미 삭제된 항목은 에러 없이 `204`로 멱등 처리
- `POST /api/v1/references/:id/restore`
  - 로그인 사용자 기준 본인 소유 항목만 대상
  - 복구 유예는 삭제 시각 기준 `24시간`
  - 유예 만료 시 `410 Gone`
  - 삭제되지 않은 항목 복구 시 `409 Conflict`

핵심 구현 파일:
- `internal/models/reference.go`
  - `deleted_by`, `restore_until` 메타데이터 추가
- `internal/handlers/references.go`
  - delete/restore 핸들러와 store 메서드 추가
- `internal/server/server.go`
  - 보호 라우트에 delete/restore 엔드포인트 연결
- `internal/handlers/references_test.go`
  - 권한, 멱등 삭제, 만료 복구, 비인증 복구, 미삭제 복구 상태 코드 테스트 추가

검증 결과:
- `$build`: `go test ./...` 통과
- `$qa`: 경계 테스트 보강 후 `go test ./...` 재통과, `VERDICT: ✅ APPROVED`

## 배운 것 (Lessons Learned)
- 현재 구조처럼 GORM soft delete가 이미 모델에 들어가 있으면, 기능을 설계할 때도 그 전제를 먼저 받아들이는 편이 맞습니다. hard delete를 새로 밀어 넣는 쪽이 오히려 더 비싸집니다.
- 삭제/복구는 UI보다 API 상태 코드 계약이 먼저 단단해야 합니다. 특히 `404`, `409`, `410`의 경계를 초기에 고정하지 않으면 이후 프론트 구현과 테스트가 쉽게 흔들립니다.
- “타인 데이터는 숨긴다”는 보안 원칙은 기능 설계 문서에서 먼저 정해 두는 편이 좋습니다. 구현 단계에서 정하려 하면 응답 코드와 QA 기대값이 계속 바뀝니다.
