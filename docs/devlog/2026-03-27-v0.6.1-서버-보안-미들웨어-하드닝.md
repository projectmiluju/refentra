# 개발 일지: 서버 보안 미들웨어 하드닝

**일자:** 2026-03-27
**관련 버전:** v0.6.1 후보
**관련 역할:** build, qa

## 배경 (Context)
운영 배포 스캐폴드를 추가한 뒤 남은 가장 큰 공백은 서버가 여전히 `middleware.CORS()` 기본 설정에 의존하고 있다는 점이었습니다. 프로덕션용 `Dockerfile`, `compose.prod`, `Caddyfile`를 추가해도 애플리케이션 레벨에서 오리진 제한과 보안 헤더가 느슨하면 운영 하드닝이 끝난 것으로 볼 수 없습니다.

## 문제 (Problem)
초기 구현 상태에서는 세 가지 문제가 있었습니다.
1. CORS가 환경별 오리진을 받지 못해 프로덕션에서 과도하게 열릴 위험이 있었습니다.
2. HSTS, CSP, `X-Frame-Options` 같은 보안 헤더가 정책적으로 관리되지 않았습니다.
3. rate limiting이 없어 인증/쓰기 API에 대한 기본 방어선이 없었습니다.

## 시도한 것들 (Attempts)
1. **Echo 기본 미들웨어만 유지:** 구현은 간단하지만 프로덕션 오리진 제한을 코드 밖에서 제어할 수 없어서 버렸습니다.
2. **보안 설정 구조 분리:** `SecurityConfig`를 두고 CORS, HSTS, CSP, rate limiting 값을 환경변수로 받도록 정리했습니다. 이 방식이 배포 스캐폴드와도 가장 잘 맞았습니다.
3. **HSTS 테스트 수정:** 첫 QA에서는 HSTS 헤더가 비어 있다고 나왔지만, 원인은 구현보다 테스트 전제가 HTTP 요청이었던 점에 있었습니다. Echo `Secure` 미들웨어는 HTTPS 전제에서 HSTS를 추가하므로, 테스트를 HTTPS 요청으로 바꿔 다시 검증했습니다.

## 최종 해결 (Resolution)
서버 보안 정책을 환경변수 기반으로 제어할 수 있도록 정리했습니다.

## 배운 것 (Lessons Learned)
- 보안 헤더 테스트는 단순 문자열 비교보다 “어떤 요청 맥락에서 헤더가 나와야 하는가”를 먼저 맞춰야 합니다. HSTS처럼 HTTPS 전제가 있는 헤더는 특히 그렇습니다.
- 인프라 스캐폴드만으로 운영 하드닝이 끝나는 것은 아닙니다. 앱 레벨의 CORS, CSP, rate limiting이 같이 맞물려야 실제 운영 기준이 됩니다.
- 이번 단계의 rate limiting은 메모리 스토어 기반이므로 단일 인스턴스에 적합합니다. 다중 인스턴스 운영을 시작하면 Redis 등 공유 스토어 기반으로 다시 설계해야 합니다.
