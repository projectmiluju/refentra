# 개발 일지: v0.11.x 대시보드 삭제/복구 UI 연결

**일자:** 2026-04-04
**관련 버전:** v0.11.x
**관련 역할:** build, qa, docs

## 배경 (Context)
`#15`에서 레퍼런스 soft delete/restore API가 먼저 정리됐지만, 실제 제품 화면에서는 아직 삭제와 복구를 수행할 수 없었습니다. 백엔드 계약만 존재하면 정책은 문서상으로만 남고, 사용자 관점에서는 여전히 "정리할 수 없는 아카이브"입니다. 이번 작업의 목표는 제품 모드 대시보드에서 삭제와 복구를 실제 상호작용으로 연결하는 것이었습니다.

## 문제 (Problem)
삭제/복구 UI는 범위를 넘기기 쉬운 작업이었습니다. 휴지통 전용 페이지, 다건 복구, 상세 상태 관리까지 넓히면 `#16`이 아니라 별도 PRD 수준으로 커집니다. 반대로 너무 단순하게 버튼만 붙이면 사용자가 24시간 복구 가능 정책을 이해하지 못하고, 삭제 후 현재 페이지가 비는 상황에서 URL/페이지네이션 정합성도 깨질 수 있습니다.

또 다른 문제는 브랜치 기준점이었습니다. 처음 `#16` 작업은 최신 `main`이 아니라 오래된 로컬 `main` 기준에서 시작돼, 문서와 상태판이 최신 `origin/main`과 쉽게 충돌할 수 있는 상태였습니다. 그래서 한 번 롤백하고, 최신 `main`을 현재 브랜치에 반영한 뒤 다시 구현했습니다.

## 시도한 것들 (Attempts)
1. **첫 시도 롤백:** 오래된 기준점에서 진행한 UI/문서 변경은 모두 되돌렸습니다. 이유는 `STATUS.md`와 devlog가 이미 `origin/main`의 `#15` 반영 상태와 충돌할 가능성이 컸기 때문입니다.
2. **기준점 재정렬 후 재구현:** 로컬 `main`을 `origin/main`으로 fast-forward 한 뒤, `feat/#16-dashboard-delete-restore-ui` 브랜치에 다시 `main`을 합쳐 기준점을 맞췄습니다. 그 다음에만 UI 작업을 다시 올렸습니다.
3. **MVP 범위 유지:** 삭제는 카드 인라인 확인 UI로, 복구는 상단 `Undo delete` 배너로 제한했습니다. 별도 휴지통 화면은 만들지 않았습니다. 이 선택으로 현재 대시보드 구조를 유지하면서도 `#16`의 완료 조건을 충족할 수 있었습니다.

## 배운 것 (Lessons Learned)
- 이슈 브랜치를 최신 `main` 기준으로 잡지 않으면, 코드 충돌보다 문서 충돌이 더 먼저 터집니다. 특히 `STATUS.md`처럼 누적형 문서는 기준점이 늦으면 바로 엇갈립니다.
- 삭제/복구 UI에서는 “삭제 자체”보다 “삭제 후 사용자가 어디로 이동하고, 무엇을 되돌릴 수 있는지”가 더 중요합니다. 페이지 보정 테스트를 별도로 둔 이유도 이 때문입니다.
- `frontend/dist/.gitkeep`는 기능 버그는 아니지만, 빌드 후 항상 삭제 상태로 남으므로 릴리즈 단계의 반복 비용을 만듭니다. 이런 종류의 잡음도 기술 부채로 기록해 두는 편이 맞습니다.
