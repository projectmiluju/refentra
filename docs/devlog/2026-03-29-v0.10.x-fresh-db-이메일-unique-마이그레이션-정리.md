# 개발 일지: fresh DB 이메일 unique 마이그레이션 정리

**일자:** 2026-03-29
**관련 버전:** v0.10.x
**관련 역할:** build, qa, ship, docs

## 배경 (Context)
최소 회원가입 인증을 PR로 묶은 뒤, GitHub Actions `Playwright Smoke E2E`가 앱 health check 단계에서 계속 실패했습니다. 로컬 개발 DB에서는 재현되지 않았지만, CI는 매번 fresh Docker volume에서 PostgreSQL을 새로 올리기 때문에 로컬과 다른 초기 스키마 경로를 탔습니다.

## 문제 (Problem)
문제의 핵심은 `users.email` unique 제약을 두 군데가 서로 다른 방식으로 관리하고 있었다는 점입니다.

- `docker/postgres/init/01-init.sql`은 `users_email_key` 제약을 가진 상태로 테이블을 만들었습니다.
- 앱 부팅 시 GORM `AutoMigrate`는 기본 naming 전략 기준으로 `uni_users_email` 제약을 기대했습니다.

그 결과 CI의 fresh DB에서는 앱 시작 직후 `ALTER TABLE "users" DROP CONSTRAINT "uni_users_email"`가 실행됐고, 실제로는 그런 제약이 없어서 migration이 실패했습니다. DB 준비가 실패하니 `/api/v1/health`는 계속 `503`을 반환했고, smoke 테스트는 시작조차 못 했습니다.

## 시도한 것들 (Attempts)
1. **제약 이름을 모델 태그에 맞추기:** `uniqueIndex:users_email_key`처럼 이름을 고정해 봤지만, fresh DB 경로에서는 여전히 GORM이 기대하는 마이그레이션 흐름과 init SQL이 완전히 일치하지 않았습니다.
2. **fresh DB 경로 로컬 재현:** 기존 volume이 남아 있는 로컬 DB에서는 문제가 숨었습니다. `docker compose down -v` 후 다시 올린 fresh volume에서 `go run .`을 실행해 CI와 같은 조건을 재현한 뒤에야 같은 오류를 로컬에서 확인할 수 있었습니다.

## 최종 해결 (Resolution)
해결 방향은 "init SQL과 AutoMigrate가 같은 역할을 두 번 하지 않게 만드는 것"이었습니다.

- init SQL에서는 `users.email`의 unique 제약을 직접 만들지 않도록 정리했습니다.
- 앱 시작 시 `main.go`에서 legacy 제약 이름 `users_email_key`를 정리하는 preflight를 추가했습니다.
- 그 뒤에 GORM `AutoMigrate`가 현재 모델 기준으로 `users.email` unique 제약을 일관되게 생성하게 했습니다.

이렇게 바꾸고 나니
- fresh DB
- 기존 volume이 남은 DB

둘 다 같은 코드 경로에서 정상 부팅했고, `/api/v1/health`가 다시 `200`으로 올라왔습니다.

## 배운 것 (Lessons Learned)
마이그레이션 이슈는 로컬에서 "이미 한 번 지나간 DB"만 보면 놓치기 쉽습니다. 특히 Docker init SQL과 ORM migration을 같이 쓰는 구조에서는, fresh volume 재현이 사실상 유일한 진실입니다. 앞으로 스키마 관련 변경은 `docker compose down -v -> up -d -> go run . -> /api/v1/health`를 기본 회귀 루틴으로 삼는 편이 맞습니다.
