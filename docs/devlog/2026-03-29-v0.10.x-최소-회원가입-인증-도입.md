# 개발 일지: 최소 회원가입 인증 도입

**일자:** 2026-03-29
**관련 버전:** v0.10.x
**관련 역할:** spec, build, qa, ship

## 배경 (Context)
포트폴리오 리디자인과 `portfolio/product` 모드 분리까지는 정리됐지만, 인증은 여전히 초기 단계의 단일 개발자 계정 전제에 머물러 있었습니다. 이 상태로는 "회원가입 가능한 실제 제품"이라는 인상을 주기 어렵고, 사용자별 데이터 소유권도 모호했습니다. 이번 사이클의 목적은 계정 관리 전체를 완성하는 것이 아니라, 최소 회원가입과 실제 계정 기반 로그인으로 제품 신뢰선을 끌어올리는 것이었습니다.

## 문제 (Problem)
문제는 세 가지였습니다.

1. 백엔드 인증이 실제 `users` 저장 구조 없이 고정 사용자 전제에 기대고 있었습니다.
2. 프론트에는 회원가입 진입점이 없어서 랜딩과 제품 모드 인증 흐름이 끊겨 있었습니다.
3. Playwright smoke를 `managed server`로 돌릴 때, 이미 떠 있는 오래된 서버가 새 `/signup` 화면을 서빙하지 않아 흰 화면이 나왔고, 실서버 smoke에서는 기본 로그인 계정이 항상 존재한다고 가정해 테스트가 깨졌습니다.

## 시도한 것들 (Attempts)
1. **실제 사용자 저장 구조로 전환:** `users` 모델, `POST /api/v1/auth/signup`, 비밀번호 해시, 이메일 unique 제약을 추가해 mock 전제를 걷어냈습니다. 회원가입 성공 시 바로 세션을 발급하도록 붙여서 가입과 로그인 사이의 불필요한 단계를 없앴습니다.
2. **로그인 화면 공유 모드화:** 로그인과 회원가입을 별도 구조로 분리하는 대신 하나의 인증 화면을 `sign in / create account` 모드로 공유시켰습니다. 이렇게 해서 랜딩의 포트폴리오 데모 CTA와 제품 모드 가입 CTA가 서로 충돌하지 않도록 했습니다.
3. **E2E 실패 원인 정리:** 처음에는 `/signup` 직접 진입이 프론트 버그처럼 보였지만, 실제 원인은 오래된 `managed server`가 새 프론트를 서빙하지 않는 테스트 방식이었습니다. 이를 확인한 뒤 새 서버를 직접 띄우는 smoke로 검증했고, 기본 로그인 계정이 없는 환경에서도 테스트가 깨지지 않도록 E2E 유틸에 기본 계정 준비 로직을 추가했습니다.

## 최종 해결 (Resolution)
최종적으로 Refentra는 최소 회원가입 기반의 실제 사용자 인증으로 전환됐습니다.

- 백엔드:
  - `users` 테이블과 `GormUserStore` 추가
  - `POST /api/v1/auth/signup` 구현
  - `email + password` 로그인 검증을 실제 저장 계정 기준으로 변경
  - 가입 성공 시 자동 로그인 세션 발급
- 프론트:
  - `/signup` 라우트 추가
  - 로그인 화면을 회원가입 모드까지 포함하는 공용 인증 화면으로 재구성
  - 랜딩에서 `Open demo`와 `Create account` 흐름을 분리
  - 가입 직후 빈 상태 대시보드 진입
- 테스트:
  - Go 테스트로 회원가입 성공, 중복 이메일, 짧은 비밀번호 검증
  - Vitest로 회원가입 클라이언트 검증과 보호 라우트 복귀 검증
  - Playwright smoke로 회원가입 후 자동 로그인, 보호 라우트, URL 복원, 저장/로그아웃 회귀 검증

## 배운 것 (Lessons Learned)
인증 같은 상태성 기능은 "코드는 맞는데 테스트 서버가 옛날 빌드를 보고 있다"는 식의 함정이 자주 생깁니다. 앞으로 브라우저 검증에서 `managed server`를 재사용할 때는 현재 코드가 실제로 반영된 서버인지 먼저 확인해야 합니다. 또 고정 테스트 계정이 있다고 가정하는 E2E는 DB 상태에 따라 쉽게 깨지므로, 테스트가 스스로 필요한 계정을 준비하도록 만드는 편이 훨씬 안정적입니다.
