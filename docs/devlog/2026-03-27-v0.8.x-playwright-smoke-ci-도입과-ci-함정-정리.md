# 개발 일지: Playwright smoke CI 도입과 CI 함정 정리

**일자:** 2026-03-27
**관련 버전:** v0.8.x
**관련 역할:** spec, build, qa, ship

## 배경 (Context)
`v0.8.0` 시점에는 Playwright 브라우저 E2E를 로컬에서 수동 실행할 수 있었지만, PR 단계에서 브라우저 회귀를 자동 차단하는 최소 신뢰선은 아직 없었습니다. 특히 로그인, 보호 라우트, 저장/재조회, URL 쿼리 복원은 사용자가 바로 체감하는 흐름이라, 코드 리뷰만으로 회귀를 막기에는 비용이 컸습니다.

## 문제 (Problem)
이번 작업은 단순히 GitHub Actions job 하나를 추가하는 수준이 아니었습니다. 실제로는 세 가지 함정이 있었습니다. 첫째, Playwright 전체 회귀를 PR마다 돌리면 비용이 커서 smoke 범위를 명시적으로 분리해야 했습니다. 둘째, CI에서 report와 trace를 남기려는 설정이 Playwright 내장 환경변수와 충돌해 산출물이 의도한 경로에 생성되지 않았습니다. 셋째, `go test ./...`가 깨끗한 체크아웃에서 `frontend/dist` 부재로 바로 실패해, Actions에서 Go 테스트가 프론트 빌드 유무에 종속되는 숨은 결합이 드러났습니다.

## 시도한 것들 (Attempts)
1. **smoke 범위를 태그와 스크립트로 분리:** 기존 Playwright 시나리오를 유지하면서 `@smoke` 태그와 `npm run test:e2e:smoke` 스크립트로 PR 차단선을 고정했습니다. 이 방식이 별도 프로젝트 분리보다 변경량이 적고 기존 테스트 구조를 덜 흔들었습니다.
2. **CI 전용 managed server 경로 추가:** 로컬 기본 경로는 `go run .`을 Playwright가 직접 띄우도록 두고, CI에서는 이미 띄운 서버를 재사용하도록 환경변수로 분기했습니다. 이렇게 해야 readiness/health check와 Playwright 실행을 분리해 실패 원인을 좁힐 수 있었습니다.
3. **HTML report 환경변수 이름 재설계:** 처음에는 `PLAYWRIGHT_HTML_REPORT=true` 같은 이름을 썼지만, 이 값이 Playwright 내장 동작과 충돌해 `frontend/true` 같은 잘못된 경로가 생겼습니다. 결국 `REFENTRA_PLAYWRIGHT_*` 접두사로 바꾸는 쪽이 안전했습니다.
4. **`go:embed` 실패 원인 추적:** 처음에는 CI 환경 문제처럼 보였지만, 실제 원인은 `main.go`의 `//go:embed all:frontend/dist`가 컴파일 시점에 실제 디렉터리를 강제한다는 점이었습니다. 빌드 산출물을 저장소에 커밋하는 대신 `frontend/dist/.gitkeep`만 추적해 embed 패턴을 만족시키는 쪽이 현재 구조에서는 가장 보수적인 해결책이었습니다.

## 최종 해결 (Resolution)
- PR용 smoke 범위를 `@smoke` 태그와 전용 npm 스크립트로 분리했습니다.
- same-repo PR 전용 `Playwright Smoke E2E` GitHub Actions job을 추가하고, concurrency, PostgreSQL/Redis readiness, 앱 health check, 아티팩트 업로드 규칙을 명시했습니다.
- Playwright CI 옵션은 `REFENTRA_PLAYWRIGHT_HTML_REPORT`, `REFENTRA_PLAYWRIGHT_TRACE_MODE`, `REFENTRA_PLAYWRIGHT_MANAGED_SERVER` 환경변수로 분리해 report 경로와 trace 정책을 안정화했습니다.
- `frontend/dist/.gitkeep`을 추적하고 `frontend/.gitignore` 예외를 추가해, 깨끗한 체크아웃에서도 `go test ./...`가 `go:embed` 때문에 실패하지 않도록 했습니다.
- README, CHANGELOG, PRD, STATUS까지 갱신해 운영 정책과 의사결정 배경이 코드 밖에서도 추적 가능하도록 정리했습니다.

## 배운 것 (Lessons Learned)
CI 자동화는 테스트를 "돌리는 것"보다, 어떤 경로가 PR 차단선인지 명시하는 작업에 더 가깝습니다. 또 환경변수 이름은 도구 내장 규칙과 충돌하지 않게 프로젝트 접두사를 붙이는 것이 안전합니다. 마지막으로 `go:embed`처럼 컴파일 시점 파일 존재를 요구하는 구조는 로컬에서는 잘 보여도 깨끗한 CI 체크아웃에서 바로 드러나므로, 테스트 가능한 최소 파일 집합을 저장소에 명시적으로 남기는 습관이 필요합니다.
