# 개발 일지: Docker PostgreSQL 개발 환경 표준화

**일자:** 2026-03-27
**관련 버전:** v0.4.0 후보
**관련 역할:** spec, build, qa, ship, docs

## 배경 (Context)
`v0.3.0`까지 레퍼런스 저장/조회 API 연동은 끝났지만, 로컬 PostgreSQL이 이미 준비돼 있다는 전제가 남아 있었습니다. 실제로 개발 환경에 DB가 없는 상태에서 저장을 검증하려 하자, 기능이 구현돼 있어도 실행 환경이 없어 바로 막히는 문제가 드러났습니다.

## 문제 (Problem)
저장소에는 DB 실행 방법이 표준화돼 있지 않았고, 서버는 외부 로컬 PostgreSQL과 하드코딩된 접속 정보를 기대하고 있었습니다. 이 상태에서는 개발자마다 준비 상태가 달라지고, DB 미연결도 빈 목록처럼 보일 수 있어 문제를 늦게 발견하게 됩니다.

## 시도한 것들 (Attempts)
1. **외부 로컬 PostgreSQL 전제 확인:** 실제 저장 실패 로그를 확인해 보니 로컬에 DB가 없거나 인증 정보가 맞지 않는 상태였습니다. 기능 버그보다 개발 환경 부재가 먼저 해결돼야 한다는 점이 명확해졌습니다.
2. **Docker 기반 DB 표준화:** DB만 Docker로 올리고 앱은 로컬에서 실행하는 방식으로 범위를 잘랐습니다. 이 선택으로 기존 개발 속도는 유지하면서도, 최소한의 재현 가능한 실행 환경을 확보했습니다.
3. **앱 시작 차단 정책 도입:** 단순히 API에서 `503`을 내는 것만으로는 부족해서, 프론트가 앱 시작 시 헬스체크를 호출하고 DB 미준비 상태면 설정 안내 페이지를 먼저 보여주도록 정리했습니다.
4. **포트 충돌 조정:** 로컬에서 다른 프로젝트가 기본 PostgreSQL 포트 `5432`를 이미 사용하고 있어, Refentra 기본 포트를 `5433`으로 조정했습니다.

## 최종 해결 (Resolution)
Docker Compose 기반 PostgreSQL 서비스를 추가하고, `.env.example`로 DB 접속 정보를 표준화했습니다. PostgreSQL 첫 생성 시 mock 사용자와 데모 레퍼런스를 넣는 init script를 연결했고, 백엔드는 `GET /api/v1/health`와 `503` 오류 처리로 DB 준비 여부를 명시적으로 노출하도록 수정했습니다. 프론트엔드는 앱 부팅 시 헬스체크를 실행해 DB가 준비되지 않았으면 로그인/대시보드 대신 설정 안내 페이지를 보여주도록 구성했습니다. QA 단계에서는 `5433` 포트 리스닝, 헬스 응답, 실제 저장/재조회까지 라이브 검증했습니다.

## 배운 것 (Lessons Learned)
로컬 개발 환경은 기능 바깥의 부속물이 아니라 제품 검증 경로의 일부입니다. 특히 1인 개발 환경에서는 “내 컴퓨터에서는 된다”는 가정이 금방 깨지므로, DB 실행 방식과 실패 시 사용자에게 보여줄 메시지까지 명세에 포함하는 편이 훨씬 안전합니다. 다음에는 기능 구현이 실제 외부 서비스나 런타임에 의존한다면, 구현보다 먼저 재현 가능한 개발 환경부터 고정하는 게 맞습니다.
