# 개발 일지: Tier 2 VPS 배포 스캐폴드 추가

**일자:** 2026-03-27
**관련 버전:** v0.6.x
**관련 역할:** ops, qa

## 배경 (Context)
`v0.6.0` 시점까지는 로컬 개발과 기능 릴리즈 흐름은 정리됐지만, 실제 운영 배포를 시작할 수 있는 자산은 없었습니다. `Go 앱 + PostgreSQL + Redis` 구조에서 가장 큰 공백은 서버를 어디에 어떤 단위로 올릴지, 그리고 배포를 수동 복붙이 아니라 어떤 파일과 절차로 재현할지 문서화되지 않았다는 점이었습니다.

## 문제 (Problem)
배포 대상은 아직 확정되지 않았지만, 운영 자산이 전혀 없으면 다음 단계에서 매번 배포 설계를 다시 해야 합니다. 특히 이 프로젝트는 Redis 세션과 PostgreSQL을 함께 사용하므로 정적 호스팅이나 단순 PaaS보다 `컨테이너 + VPS` 구성이 자연스러운데, 이를 뒷받침하는 `Dockerfile`, 프로덕션 compose, HTTPS 프록시, CI/CD 초안이 없는 상태였습니다.

## 시도한 것들 (Attempts)
1. **Tier 1 PaaS 관점 검토:** 프론트와 API를 분리 배포하는 방향을 먼저 검토했지만, 현재 구조는 Go 앱이 프론트 자산까지 함께 서빙하고 Redis 세션도 사용합니다. 이 조합에서는 서비스 분리가 오히려 운영 복잡도를 늘린다고 판단했습니다.
2. **AWS 직접 진입 검토:** EC2/RDS/ElastiCache 조합도 가능했지만, 현재 단계의 앱 규모에 비해 선택지가 너무 많고 초기 운영 비용이 큽니다. 결국 `Tier 2: 단일 VPS + Docker Compose`가 현재 프로젝트 성숙도에 더 맞는다고 정리했습니다.
3. **운영 자산만 먼저 추가:** 실제 배포는 보류하고, `Dockerfile`, `docker-compose.prod.yml`, `Caddyfile`, GitHub Actions 워크플로우, `.env.production.example`만 먼저 추가했습니다. 이렇게 하면 서버 구매와 시크릿 입력이 늦어져도 다음 단계에서 바로 실행 가능한 출발점을 확보할 수 있습니다.

## 최종 해결 (Resolution)
운영 토폴로지를 `Hetzner VPS + Docker Compose + Caddy + 내부 PostgreSQL/Redis`로 고정하고, 프로덕션 스캐폴드를 저장소에 추가했습니다.

- `docker build -t refentra:qa .`

둘 다 통과했고, 적어도 “파일이 존재한다” 수준이 아니라 실제 이미지 빌드와 compose 해석이 가능한 상태까지는 확보했습니다.

## 배운 것 (Lessons Learned)
- 운영 스캐폴드는 실제 서버가 없어도 먼저 저장소에 고정해 두는 편이 낫습니다. 나중에 서버를 산 뒤 설계를 시작하면 인프라 결정이 다시 흔들립니다.
- 인프라 파일만 추가한다고 운영 준비가 끝나는 것은 아닙니다. CORS 하드닝, GitHub Actions 실제 런너 검증, 시크릿 주입 절차, 백업 정책은 별도 작업입니다.
- 배포 자동화에서도 “실행 가능한 최소 단위”를 먼저 확보하는 게 중요합니다. 이번 사이클은 실제 배포보다 재현 가능한 출발점 확보에 의미가 있습니다.
