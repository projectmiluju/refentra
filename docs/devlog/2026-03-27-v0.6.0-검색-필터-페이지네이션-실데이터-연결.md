# 개발 일지: 검색, 필터, 페이지네이션 실데이터 연결

**일자:** 2026-03-27
**관련 버전:** v0.6.0 후보
**관련 역할:** spec, build, qa

## 배경 (Context)
Refentra는 저장/조회와 인증, 브라우저 E2E까지 갖췄지만 저장된 레퍼런스를 다시 찾는 기능은 여전히 목업 수준이었습니다. 레퍼런스 수가 늘어나기 시작하면 전체 목록을 눈으로 훑는 방식은 바로 비효율이 되기 때문에, 검색과 태그 필터, 페이지네이션을 실제 데이터 기준으로 연결할 필요가 있었습니다.

## 문제 (Problem)
기존 대시보드는 검색 입력, 태그 배지, 페이지 번호를 보여주기만 하고 실제 서버 조회에는 사용하지 않았습니다. 백엔드도 `GET /api/v1/references`에서 단순 정렬과 `Limit(50)`만 수행하고 있어, 데이터가 많아질수록 사용자가 원하는 레퍼런스를 빠르게 좁혀갈 방법이 없었습니다.

## 시도한 것들 (Attempts)
1. **클라이언트 전역 필터링 검토:** 이미 받아온 목록을 프론트에서만 필터링하는 방식도 가능했지만, 데이터가 늘어날수록 페이지네이션과 정합성이 깨질 위험이 커서 채택하지 않았습니다.
2. **서버 주도 조회로 방향 고정:** 검색, 태그, 페이지를 모두 `GET /api/v1/references` 쿼리로 처리하고 프론트는 서버 응답 메타데이터를 렌더링하는 방식으로 정리했습니다.
3. **테스트 계약 재정비:** 응답 구조가 `items + total_count + total_pages + available_tags`로 바뀌면서 기존 Dashboard 테스트 fixture가 모두 깨졌습니다. 저장 후 재조회까지 포함한 새 계약으로 fixture를 재작성해 회귀를 정리했습니다.

## 최종 해결 (Resolution)
백엔드 핸들러는 `search`, `tags`, `page`, `limit`를 파싱해 검색/태그/페이지네이션을 실제로 처리하고, 응답에 페이지 메타데이터와 사용 가능한 태그 목록을 담도록 확장했습니다. 프론트 대시보드는 검색 디바운스, 태그 토글, 페이지 이동을 서버 조회에 연결했고, 일반 빈 상태와 검색 결과 없음 상태를 분리했습니다. 테스트는 Vitest와 Go 테스트를 새 계약에 맞게 보강했고, 기존 Playwright 핵심 플로우도 회귀 없이 통과하는지 다시 확인했습니다.

## 배운 것 (Lessons Learned)
검색/필터/페이지네이션은 UI 요소를 추가하는 문제가 아니라 조회 계약을 다시 정의하는 문제에 가깝습니다. 특히 목록 저장 기능이 이미 있는 상태에서는 “저장 후 현재 조건과 맞는가”를 어떻게 정합성 있게 반영할지까지 같이 생각해야 합니다. 또한 목록 API 계약이 바뀌면 단위 테스트 fixture를 먼저 새 구조에 맞게 재정렬해야 이후 수정 비용이 줄어듭니다.
