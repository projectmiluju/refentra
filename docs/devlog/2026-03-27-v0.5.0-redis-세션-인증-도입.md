# 개발 일지: Redis 세션 인증 도입

**일자:** 2026-03-27
**관련 버전:** v0.5.0 후보
**관련 역할:** spec, build, qa, ship, docs

## 배경 (Context)
`v0.4.0`까지는 저장/조회와 Docker 개발 환경은 정리됐지만, 로그인은 여전히 프론트엔드 `localStorage` 플래그에 의존하고 있었습니다. 이 구조에서는 서버가 사용자를 검증하지 못하고, 보호된 API도 사실상 “로그인한 척”만 하면 호출 가능한 상태였습니다.

## 문제 (Problem)
현재 구조의 가장 큰 약점은 인증 상태가 서버가 아니라 브라우저 임의 상태에 달려 있다는 점이었습니다. 이 상태에서는 `uploader_id`도 실제 사용자 기반이 아니고, 세션 만료/로그아웃/재발급 같은 핵심 인증 정책을 검증할 방법도 없습니다.

## 시도한 것들 (Attempts)
1. **사용자 계정 체계까지 한 번에 넣는 방안 검토:** 실제 계정, 비밀번호 저장, 권한 체계를 한 번에 넣는 것은 범위가 너무 커서 이번 사이클의 핵심 문제를 늦출 가능성이 컸습니다.
2. **mock 사용자 유지 + 실제 세션 전환:** 범위를 줄이기 위해 mock 사용자 1명은 유지하고, 세션 발급/검증만 서버 중심으로 바꾸는 방향으로 잘랐습니다. 이렇게 하면 제품 리스크를 빨리 줄이면서도 이후 실제 계정 체계로 확장할 수 있습니다.
3. **토큰 저장 방식 조정:** 초기에 브라우저 저장소 활용도 검토했지만, 최종적으로는 Access Token을 `httpOnly` 쿠키로, Refresh Token은 Redis로 관리하는 쪽으로 정리했습니다. 이 선택으로 현재 구조에서 XSS 노출을 줄이면서도 리이슈 정책을 구현할 수 있었습니다.
4. **회전 정책 검증:** Refresh Token은 rotating refresh token으로 설계했고, QA에서 old refresh 재사용이 실제로 `SESSION_EXPIRED`로 막히는지 순차적으로 검증했습니다.

## 최종 해결 (Resolution)
Docker Compose에 Redis를 추가하고, 백엔드에 `internal/auth` 패키지를 만들어 JWT 쿠키 발급, Redis Refresh Token 저장, 인증 미들웨어를 분리했습니다. 프론트는 `localStorage` 인증을 제거하고, 앱 부팅 시 `auth/me`로 로그인 상태를 확인하며, 세션 만료 시 리이슈 또는 로그인 페이지 복귀로 흐름을 바꿨습니다. 레퍼런스 API는 인증 사용자 기준으로 보호되며, 빈 상태는 첫 로그인 사용자 기준 온보딩 가이드로 바뀌었습니다. QA 단계에서는 `login → me → references → refresh rotation → logout → me` 전체를 라이브로 확인했습니다.

## 배운 것 (Lessons Learned)
“실제 인증”은 반드시 사용자 계정 시스템 전체를 뜻하지는 않습니다. 현재처럼 가장 큰 제품 리스크가 “서버가 사용자를 모른다”는 문제일 때는, mock 사용자를 유지하더라도 세션 검증을 서버로 옮기는 것이 훨씬 효과적입니다. 또 인증/세션 같은 상태성 기능은 브라우저 단위 테스트만으로는 부족하고, 쿠키와 저장소가 실제로 교차하는 라이브 검증이 꼭 필요합니다.
