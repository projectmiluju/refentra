# 개발 일지: Playwright 기반 브라우저 E2E 도입

**일자:** 2026-03-27
**관련 버전:** v0.5.1 후보
**관련 역할:** spec, build, qa

## 배경 (Context)
`v0.5.0`까지 로그인, 실제 세션 인증, 레퍼런스 저장/조회는 구현됐지만 검증 수단은 `curl`, Vitest, Go 테스트에 머물러 있었습니다. 이 상태에서는 브라우저 리다이렉트, `httpOnly` 쿠키 세션, 새로고침 후 재조회 같은 실제 사용자 흐름이 깨져도 늦게 발견될 위험이 컸습니다.

## 문제 (Problem)
가장 큰 공백은 "핵심 플로우가 실제 브라우저에서 끝까지 유지되는가"를 자동으로 보장하지 못한다는 점이었습니다. 특히 비로그인 상태에서의 보호 URL 접근, 로그인 후 대시보드 진입, 저장 후 새로고침 재조회, 로그아웃 뒤 재접근 차단은 단위 테스트만으로 충분히 커버되지 않았습니다.

## 시도한 것들 (Attempts)
1. **Vitest 재사용 검토:** 기존 테스트 러너 위에서 브라우저 흐름까지 해결할 수 있는지 봤지만, 실제 쿠키/리다이렉트/새로고침 검증에는 한계가 분명했습니다.
2. **Playwright 최소 생존 버전 채택:** 우선 Chromium 하나와 핵심 시나리오 3개만 넣는 방향으로 범위를 줄였습니다. 이 선택 덕분에 인프라 과투자 없이 회귀선을 빠르게 확보할 수 있었습니다.
3. **Vitest 충돌 정리:** 초기에는 Vitest가 `e2e` 폴더와 `node_modules` 내부 테스트를 잘못 수집해 실패했습니다. 테스트 범위를 `src/**/*.{test,spec}.{ts,tsx}`로 고정해 러너 충돌을 정리했습니다.
4. **실브라우저 실행 검증:** 샌드박스에서는 로컬 포트 바인딩과 Docker DB/Redis 연결이 막혀 Playwright 실행이 실패했습니다. 권한 상승으로 다시 실행해 실제 브라우저 3개 시나리오 통과를 확인했습니다.

## 최종 해결 (Resolution)
Playwright를 프론트엔드 개발 의존성으로 추가하고, `frontend/playwright.config.ts`에서 `go run .` 기반 `webServer`를 사용하도록 구성했습니다. 브라우저 시나리오는 보호 URL 리다이렉트, 온보딩 빈 상태, 저장 후 새로고침 재조회와 로그아웃까지 포함하도록 만들었습니다. 문서에는 실행 방법과 범위를 기록했고, README와 STATUS도 새로운 테스트 체계에 맞게 업데이트했습니다.

## 배운 것 (Lessons Learned)
브라우저 E2E는 "테스트를 더 많이 만든다"보다 "어떤 회귀를 실제로 막을지 좁게 정한다"가 더 중요했습니다. 이번처럼 핵심 플로우 3개만 먼저 자동화해도 품질 체감이 크게 올라갑니다. 또, 새 테스트 도구를 넣을 때는 기존 러너의 수집 범위를 먼저 고정하지 않으면 의도치 않은 실패가 바로 생깁니다.
