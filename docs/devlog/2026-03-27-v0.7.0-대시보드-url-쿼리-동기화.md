# 개발 일지: 대시보드 URL 쿼리 동기화

**일자:** 2026-03-27
**관련 버전:** v0.7.0
**관련 역할:** spec, build, qa

## 배경 (Context)
`v0.6.0`에서 검색, 태그 필터, 페이지네이션이 실제 데이터와 연결됐지만, 상태가 전부 컴포넌트 내부 `useState`에만 머물렀습니다. 이 구조에서는 새로고침, 공유 URL, 뒤로가기, 로그인 후 복귀 상황에서 사용자가 보던 검색 상태를 복원할 수 없었습니다.

## 문제 (Problem)
핵심 사용자 가치가 "저장한 레퍼런스를 다시 빠르게 찾는 것"인데, 찾는 상태를 URL로 표현하지 못하면 같은 검색 맥락을 다시 여는 경험이 끊깁니다. 특히 보호된 `/dashboard` 경로 특성상 비로그인 사용자가 쿼리 포함 URL로 진입했을 때 로그인 후 같은 상태로 돌아가야 하는 요구가 생겼습니다.

## 시도한 것들 (Attempts)
1. **대시보드 내부 상태만 URL로 일방향 복사**
   - 검색/태그/페이지 변경 시 URL만 갱신하고, 초기 상태 복원은 따로 두는 접근을 먼저 봤습니다.
   - 하지만 새로고침과 뒤로가기를 처리하려면 결국 URL이 단일 진실 공급원이어야 해서 유지하지 않았습니다.
2. **프론트만 반복 `tags` 파라미터로 바꾸기**
   - `tags=A&tags=B` 직렬화만 먼저 바꾸면 될 것처럼 보였지만, 서버는 기존 `tags=a,b`만 파싱하고 있었습니다.
   - 공유 URL에서 실제 필터가 깨지므로 백엔드 파싱도 같이 수정해야 했습니다.
3. **URL 복원 자체를 Playwright E2E로 추가**
   - 실브라우저에서도 바로 묶으려 했지만, 현재 라우팅/로그인 리다이렉트 검증은 불안정했습니다.
   - 이번 사이클은 Vitest에서 URL 복원과 로그인 후 복귀를 먼저 고정하고, 브라우저 레벨 확장은 후속 작업으로 남겼습니다.

## 최종 해결 (Resolution)
`frontend/src/lib/dashboardQuery.ts`를 추가해 `search`, `tags`, `page`를 URL 쿼리의 단일 상태로 정의했습니다. 대시보드는 `useSearchParams`를 기준으로 초기 상태를 복원하고, 검색/태그/페이지 변경 시 URL을 먼저 갱신한 뒤 서버 조회를 수행합니다.

잘못된 `page` 쿼리는 단순 무시하지 않고 안내 메시지를 보여준 뒤 마지막 유효 페이지 또는 `1페이지`로 보정했습니다. 동시에 `frontend/src/lib/references.ts`와 `internal/handlers/references.go`를 같이 바꿔 반복 `tags` 파라미터와 기존 콤마 구분 형식을 모두 받을 수 있게 정리했습니다.

로그인 흐름은 `App.tsx`와 `Login.tsx`에서 복귀 경로를 유지하도록 보강했습니다. 비로그인 상태로 `/dashboard?...`에 접근해도, 로그인 후 같은 대시보드 상태로 돌아가도록 Vitest에서 검증했습니다.

## 배운 것 (Lessons Learned)
- URL이 사용자 상태의 일부라면, 컴포넌트 상태가 아니라 URL 자체를 단일 진실 공급원으로 두는 편이 낫습니다.
- 프론트 URL 포맷을 바꿀 때는 직렬화만 보지 말고, 백엔드 파싱 계약까지 함께 바꿔야 실제 공유 링크가 깨지지 않습니다.
- 이번처럼 라우팅과 인증이 섞인 경우, Playwright에 바로 모든 시나리오를 얹기보다 Vitest로 계약을 먼저 고정한 뒤 브라우저 시나리오를 확장하는 편이 더 안정적입니다.
