# 개발 일지: 포트폴리오 리디자인과 모드 분리 정리

**일자:** 2026-03-29
**관련 버전:** v0.9.x
**관련 역할:** ui, build, qa, ship, docs

## 배경 (Context)
`v1.8` PRD의 목표는 단순한 스타일 교체가 아니라, Refentra를 포트폴리오 제출용 랜딩과 실제 제품형 대시보드로 다시 읽히게 만드는 것이었습니다. 특히 랜딩의 인상, 대시보드의 완성도, 레퍼런스 추가 모달의 제출 흐름까지 한 덩어리로 보여야 했고, 이후에는 이 결과를 GitHub 이슈 `#5`, `#6`, `#7`, `#8` 기준으로 다시 정리해야 했습니다.

## 문제 (Problem)
이번 사이클의 실제 문제는 화면을 새로 만드는 것보다 “정책과 구현을 다시 맞추는 일”에 가까웠습니다. 첫째, 초기 리디자인은 빠르게 `main`에 반영됐지만 원래 팀 정책인 `issue -> branch -> PR` 흐름을 따르지 못했습니다. 둘째, 시각 리뉴얼만으로는 `#5`가 요구한 `portfolio mode / product mode` 분리가 충족되지 않았고, `#8`도 클라이언트 측 제출 잠금만으로는 서버 기준 중복 제출 방지까지 끝났다고 보기 어려웠습니다. 셋째, 리디자인 문서도 “문서가 기준”이 아니라 실제 Stitch 산출물과 최종 구현을 기준으로 다시 해석해야 했습니다.

## 시도한 것들 (Attempts)
1. **Stitch 산출물을 문서 기준으로 강제:** 처음에는 문서에 정의한 토큰과 카피를 Stitch에 밀어 넣는 쪽으로 접근했습니다. 하지만 실제 결과물은 Stitch의 스타일 시스템과 충돌했고, 시각적으로도 어정쩡했습니다.
2. **UI 전체를 영어 기반 cold minimal 톤으로 재설계:** 랜딩, 로그인, 대시보드, 모달을 영어 중심으로 다시 구성하고, 공통 토큰과 카피도 전면 수정했습니다. 이 선택으로 포트폴리오용 인상은 개선됐지만, `portfolio mode` 분리와 서버 보호 정책 같은 이슈 수준 요구사항은 남았습니다.
3. **이슈 기준 재정리:** `#6`, `#7`은 이미 `main`에 들어간 커밋을 근거로 수동 정리하고, 남은 `#5`, `#8`은 새 브랜치 `feat/issue-5-8-portfolio-mode-guard`에서 다시 마무리했습니다.
4. **프론트/백 분리 보강:** 프론트는 `/dashboard?mode=portfolio`에서 샘플 데이터 레이어를 타도록 분기했고, 백엔드는 `POST /api/v1/references` 저장 전에 동일 사용자 기준 `url + title` 중복을 검사하는 가드를 추가했습니다.

## 최종 해결 (Resolution)
- 랜딩은 cold minimal 톤으로 재설계하고, 데모 CTA를 `/dashboard?mode=portfolio`로 연결했습니다.
- 대시보드에는 `portfolio`와 `product` 모드를 명시적으로 분리해, 포트폴리오에서는 샘플 데이터 기반 검색, 태그 필터, 페이지네이션, 모달 저장이 동작하도록 만들었습니다.
- 제품 모드의 기존 API 조회/저장 계약은 유지했습니다.
- 서버 저장 경로에는 동일 사용자 기준 중복 생성 방지 가드를 추가해 `#8`의 서버 측 제출 보호까지 닫았습니다.
- 테스트는 Vitest, Go 테스트, Playwright smoke까지 새 정책에 맞춰 보강했습니다.
- `#6`, `#7`은 기존 `main` 직행 커밋 기준으로 이슈 코멘트와 함께 닫았고, `#5`, `#8`은 별도 브랜치와 PR `#9`를 통해 머지했습니다.

## 배운 것 (Lessons Learned)
리디자인 작업은 화면만 바꾸면 끝나지 않습니다. 포트폴리오 제출용 화면과 실제 제품 모드가 한 코드베이스에 공존한다면, 시각 언어보다 먼저 “어떤 경로가 데모고 어떤 경로가 실제 데이터인가”를 명시적으로 분리해야 합니다. 또 정책을 한번 어기고 `main`에 직접 반영하면, 나중에 이슈/PR 이력을 다시 정리하는 비용이 생각보다 큽니다. 앞으로는 UI 리디자인 작업도 반드시 이슈 단위 브랜치에서 시작해야 추적성이 유지됩니다.
