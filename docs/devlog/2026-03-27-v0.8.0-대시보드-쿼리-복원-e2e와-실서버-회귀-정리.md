# 개발 일지: 대시보드 쿼리 복원 E2E와 실서버 회귀 정리

**일자:** 2026-03-27
**관련 버전:** v0.8.0
**관련 역할:** build, qa

## 배경 (Context)
`v0.7.0`에서 대시보드 검색 상태를 URL 쿼리와 동기화했지만, 그 복원 흐름은 Vitest 중심으로만 검증되고 있었습니다. 직접 URL 진입, 로그인 후 복귀, 새로고침 유지 같은 브라우저 레벨 동작은 아직 자동 보증이 없었습니다.

## 문제 (Problem)
새 Playwright 시나리오를 붙이는 과정에서 두 종류의 문제가 드러났습니다. 첫째, Playwright가 기존 `8080` 서버를 재사용하면서 현재 코드가 아닌 오래된 서버를 대상으로 테스트하는 상황이 있었습니다. 둘째, 실서버 기준 `GET /api/v1/references`가 태그 목록 조회 SQL 때문에 500을 반환하고 있었고, 이 회귀가 기존 auth-flow E2E까지 함께 깨뜨렸습니다.

## 시도한 것들 (Attempts)
1. **URL 복원 전용 spec 분리:** 기존 `auth-flow.spec.ts`에 계속 책임을 몰지 않고 `dashboard-query.spec.ts`와 mock 헬퍼를 분리했습니다. 이 선택으로 URL 상태 복원과 인증 기본 흐름을 분리해 읽기 쉬운 구조를 확보했습니다.
2. **대시보드 쿼리 읽기 경로 단순화:** `Dashboard`가 `window.location`과 `location.search`를 섞어 읽던 경로를 정리하고, React Router의 `location.search`를 기준으로 쿼리를 해석하도록 수정했습니다.
3. **Playwright 환경 추적:** 처음에는 앱 버그처럼 보였지만, trace와 실제 `8080` 프로세스를 확인해 보니 오래 떠 있던 서버가 재사용되고 있었습니다. 결국 `reuseExistingServer`를 끄는 쪽이 테스트 신뢰성 측면에서 더 맞았습니다.
4. **실서버 API 확인:** auth-flow 회귀를 따라가며 실제 `GET /api/v1/references`를 호출해 보니, `references` 테이블 raw SQL이 PostgreSQL 예약어 충돌로 실패하고 있음을 확인했습니다.

## 최종 해결 (Resolution)
- Playwright에 URL 복원 전용 E2E를 추가했습니다.
- Playwright 설정에서 기존 서버 재사용을 막고, 매 실행마다 현재 코드 기준 `go run .` 서버를 띄우도록 바꿨습니다.
- `GET /api/v1/references`의 태그 목록 조회 SQL을 `FROM "references"`로 고쳐 실서버 500 회귀를 제거했습니다.
- 기존 auth-flow E2E도 현재 로그인 리다이렉트 쿼리 정책과 references 응답 계약에 맞게 정리했습니다.

## 배운 것 (Lessons Learned)
브라우저 E2E가 실패할 때는 앱 코드만 의심하면 안 됩니다. 실행 대상 서버가 현재 코드인지, 테스트 환경이 실제 계약을 따르는지 먼저 확인해야 합니다. 또 E2E를 추가하면 기존 회귀선이 숨기고 있던 실서버 버그가 함께 드러날 수 있으므로, 새 시나리오와 기존 시나리오를 끝까지 같이 돌리는 습관이 필요합니다.
