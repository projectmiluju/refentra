# [PRD] 레퍼런스 검색, 태그 필터, 페이지네이션

**Status:** Approved
**Date:** 2026-03-27
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** `v0.5.1` 기준 Refentra는 로그인, 저장, 재조회, 브라우저 E2E까지 확보했지만 저장된 레퍼런스를 다시 찾는 기능은 아직 목업 수준입니다. 레퍼런스 수가 늘어날수록 목록 전체를 눈으로 훑는 방식은 바로 한계에 부딪히므로, 검색과 태그 필터, 번호 기반 페이지네이션을 실제 데이터 기준으로 완성해야 합니다.
- **타겟 워크플로우:** 사용자가 대시보드에 들어와 키워드를 입력하거나 태그를 선택하면 원하는 레퍼런스만 빠르게 좁혀지고, 결과가 많을 때는 페이지 번호를 눌러 다음 묶음으로 이동할 수 있어야 합니다.
- **⚠️ 가정 (Assumption):**
  - 검색 기본 대상은 `title + description + url` 통합 키워드입니다.
  - 태그 필터는 다중 선택 OR 조건이 아니라, 선택된 태그를 모두 포함해야 하는 AND 조건으로 설계합니다.
  - 페이지네이션은 무한 스크롤이 아니라 기존 UI와 맞는 번호 기반(page number) 방식으로 고정합니다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] `GET /api/v1/references`가 `page`, `limit`, `search`, `tags` 쿼리를 실제로 처리해야 한다.
- [ ] 대시보드 검색 입력은 디바운스 후 서버 조회를 다시 호출해야 한다.
- [ ] 태그를 클릭하면 활성/비활성 상태가 바뀌고, 선택된 태그 조합으로 서버 조회를 다시 호출해야 한다.
- [ ] 검색과 태그 필터는 동시에 적용되어야 한다.
- [ ] 결과 수가 `limit`를 초과하면 페이지 번호 목록을 보여주고, 페이지 변경 시 해당 페이지 결과를 다시 불러와야 한다.
- [ ] 검색/필터 결과가 0건이면 일반 빈 상태와 구분되는 “검색 결과 없음” 상태를 보여줘야 한다.
- [ ] 조회 실패 시 자동 재시도 없이 에러 메시지와 `다시 시도` 버튼을 유지해야 한다.
- [ ] 레퍼런스를 새로 저장한 뒤 현재 검색/필터 조건에 맞는 경우만 현재 목록에 반영하거나, 조건과 맞지 않으면 전체 재조회로 정합성을 맞춰야 한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**
  - 초기 진입: `Idle -> Loading -> Success | Empty | Error`
  - 검색 입력: `Success -> Searching -> Success | Empty | Error`
  - 태그 선택: `Success -> Filtering -> Success | Empty | Error`
  - 페이지 변경: `Success -> Paginating -> Success | Error`
- **핵심 데이터 필드:**
  - 클라이언트 조회 상태:
    - `searchQuery: string`
    - `selectedTags: string[]`
    - `page: number`
    - `pageSize: number`
    - `totalCount: number`
    - `totalPages: number`
  - 서버 응답 구조:
    - `items: Reference[]`
    - `page: number`
    - `limit: number`
    - `total_count: number`
    - `total_pages: number`
    - `available_tags: string[]`
- **API 엔드포인트 (예상):**
  - `GET /api/v1/references?page=1&limit=10&search=react&tags=Frontend,Design`
  - `POST /api/v1/references`
- **서버 쿼리 정책:**
  - `search`는 `title`, `description`, `url`에 대해 부분 일치 검색
  - `tags`는 쉼표 구분 문자열 또는 반복 쿼리 중 하나로 통일
  - 정렬 기준은 기존과 동일하게 `created_at desc`

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| 검색 결과 0건 | 일반 온보딩이 아니라 별도 “검색 결과 없음” 메시지와 검색 초기화 액션 표시 | P0 |
| 태그/검색 조합 결과 0건 | 선택된 조건을 유지한 채 결과 없음 상태 표시 | P0 |
| 서버 `500`/`503` | 자동 재시도 없이 에러 메시지와 `다시 시도` 버튼 표시 | P0 |
| 검색 입력을 빠르게 연속 입력 | 프론트는 디바운스 처리, 가장 마지막 입력 기준 결과만 반영 | P0 |
| 페이지 이동 중 다른 검색어 입력 | 이전 요청 결과를 무시하고 최신 검색 조건으로 1페이지부터 다시 조회 | P0 |
| 저장 버튼 중복 클릭 | 기존 정책 유지, 중복 제출 방지 | P1 |
| 선택한 페이지가 총 페이지 수보다 커짐 | 검색/필터 변경 시 현재 페이지를 1로 리셋 | P0 |
| 태그가 하나도 없는 데이터만 존재 | 태그 영역은 비어 있어도 레이아웃이 깨지지 않아야 함 | P1 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** 접근법 A, 기존 API 확장 + 서버 주도 페이지네이션
- **핵심 기술:**
  - **프론트엔드:** 기존 React/Vite 유지
    - 근거: 현재 대시보드 상태 관리와 E2E가 이미 존재하므로, 검색/필터/페이지 상태만 확장하는 편이 가장 빠릅니다.
  - **백엔드:** 기존 Go/Echo/GORM 유지
    - 근거: 검색과 페이지네이션은 서버 쿼리와 응답 구조가 핵심이므로 현재 핸들러를 확장하는 것이 자연스럽습니다.
  - **DB:** 기존 PostgreSQL 유지
    - 근거: 부분 일치 검색과 태그 JSON/배열 처리 모두 현재 스택 안에서 해결 가능합니다.
  - **기존 프로젝트 스택과의 호환성:** 호환

### 접근법 A: 최소 생존 버전
- **요약:** 현재 `GET /api/v1/references`를 확장해 검색/태그/페이지네이션을 한 번에 처리하고, 대시보드는 서버 응답을 그대로 렌더링합니다.
- **난이도:** M
- **리스크:** 낮음
- **장점:**
  - 지금 부족한 핵심 제품 가치를 가장 빠르게 메웁니다.
  - 기존 저장/조회 구조와 E2E를 크게 흔들지 않습니다.
  - 데이터가 늘어나도 클라이언트가 전체를 들고 있지 않아도 됩니다.
- **단점:**
  - 검색 성능 최적화는 이후 인덱스/전문 검색 도입 전까지 제한적입니다.
  - 태그 집계와 페이지 메타데이터 응답 설계가 추가됩니다.

### 접근법 B: 이상적 아키텍처
- **요약:** 검색 전용 인덱스 전략, 태그 집계 API 분리, 커서 기반 페이지네이션까지 포함해 장기 확장성을 먼저 설계합니다.
- **난이도:** L
- **리스크:** 중간
- **장점:**
  - 수천~수만 건 규모에도 구조적으로 더 안정적입니다.
  - 전문 검색, 정렬, 추천 확장까지 고려하기 쉽습니다.
  - 태그/검색 로직을 더 명확히 분리할 수 있습니다.
- **단점:**
  - 현재 프로젝트 규모 대비 과합니다.
  - 1인 개발 기준으로 구현과 테스트 시간이 급격히 늘어납니다.
  - 제품 가치 검증보다 인프라 설계 비중이 커집니다.
