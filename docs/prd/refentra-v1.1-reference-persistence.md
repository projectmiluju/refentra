# [PRD] Refentra v1.1 레퍼런스 실제 저장/조회 연동

**Status:** Approved
**Date:** 2026-03-26
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** 현재 Refentra는 로그인 보호와 레퍼런스 등록 UI는 존재하지만, 프론트엔드가 로컬 상태만 갱신해 "저장된 것처럼 보이지만 실제로는 저장되지 않는" 착시가 있습니다. 이번 사이클의 목표는 레퍼런스가 실제 백엔드 API와 DB를 통해 저장되고, 대시보드 재진입 및 새로고침 후에도 동일하게 조회되도록 만들어 제품의 최소 신뢰성을 확보하는 것입니다.
- **타겟 워크플로우:** 사용자가 대시보드에서 레퍼런스를 추가하면 서버에 저장되고, 저장 성공 피드백을 받은 뒤 목록 상단에서 즉시 확인할 수 있어야 합니다. 이후 페이지를 새로고침하거나 다시 접속해도 저장된 레퍼런스가 유지되어야 합니다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] 대시보드 최초 진입 시 프론트엔드는 `GET /api/v1/references`를 호출해 실제 데이터를 렌더링해야 한다.
- [ ] 레퍼런스 추가 모달의 저장 버튼은 `POST /api/v1/references`를 호출해야 한다.
- [ ] 저장 성공 시 서버가 반환한 레퍼런스를 기준으로 목록을 갱신해야 한다.
- [ ] 저장 실패 시 사용자는 실패 원인을 이해할 수 있는 에러 메시지를 봐야 하며, 입력값은 유지되어야 한다.
- [ ] 저장 중에는 중복 제출이 발생하지 않도록 저장 버튼을 비활성화해야 한다.
- [ ] 대시보드는 최소 `Loading / Empty / Success / Error` 4개 상태를 가져야 한다.
- [ ] 새로고침 후에도 직전에 저장한 레퍼런스가 다시 조회되어야 한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**
  - 대시보드 조회: `Idle -> Loading -> Success | Empty | Error`
  - 레퍼런스 저장: `Idle -> Submitting -> Success | Error`
- **핵심 데이터 필드:**
  - `id`: 서버가 부여한 식별자
  - `url`: 문자열, trim 후 저장
  - `title`: 문자열, trim 후 저장
  - `description`: 문자열, trim 후 저장
  - `tags`: 문자열 배열
  - `uploader_id`: 현재는 mock 사용자 `user-1234`
  - `created_at`: 서버 저장 시각
- **프론트엔드 상태 모델:**
  - `references: ReferenceItem[]`
  - `isLoading: boolean`
  - `loadError: string`
  - `isSubmitting: boolean`
  - `submitError: string`
- **API 엔드포인트 (예상):**
  - `GET /api/v1/references`
    - 응답: 최신순 레퍼런스 배열
    - 성공: `200 OK`
  - `POST /api/v1/references`
    - 요청: `{ url, title, description, tags }`
    - 성공: `201 Created`
    - 실패: `400 Bad Request`, `500 Internal Server Error`

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| 대시보드 첫 진입 시 데이터 0건 | 빈 목록 대신 "첫 레퍼런스를 추가해 아카이브를 시작"하는 Empty State 표시 | P0 |
| 조회 API 실패 | 목록 영역에 에러 메시지와 `다시 시도` 버튼 표시 | P0 |
| 저장 API가 400 반환 | 모달 내부에 서버 메시지 또는 정형화된 검증 실패 메시지 표시, 입력값 유지 | P0 |
| 저장 API가 500 반환 | 모달 내부에 저장 실패 메시지 표시, 재시도 허용 | P0 |
| 저장 버튼 연속 클릭 | 저장 중 버튼 비활성화, 프론트엔드에서 중복 제출 차단 | P0 |
| 비로그인 사용자가 대시보드 직접 접근 | 현재 정책 유지: 로그인 페이지로 리다이렉트 | P1 |
| 저장 성공 직후 조회 목록과 순서가 어긋남 | `POST` 응답으로 받은 서버 객체를 기준으로 목록 상단 삽입 | P1 |
| 서버 DB 연결 실패 상태로 조회/저장 | 현재 서버 동작을 고려해 에러 UI를 명확히 표시하고, 성공처럼 보이는 fallback은 금지 | P0 |
| 삭제/복구 요구 | 이번 사이클 범위에서 제외 | P2 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** 기존 Go API + React 프론트엔드 직접 연동
- **핵심 기술:**
  - **프론트엔드:** React 18 + Vite
    - 근거: 현재 화면과 상태 관리가 이미 React로 구성되어 있어 가장 적은 변경으로 API 연동 가능
  - **백엔드:** Go + Echo
    - 근거: `GET/POST /api/v1/references`가 이미 존재하므로 계약을 유지하면서 응답만 안정화하면 됨
  - **DB:** PostgreSQL + GORM
    - 근거: 기존 모델과 마이그레이션 흐름이 이미 이 스택을 전제로 함
  - **테스트:** Vitest + Go test
    - 근거: 현재 검증 체계에 그대로 얹을 수 있음
- **기존 프로젝트 스택과의 호환성:** 호환

### 접근법 A: 최소 생존 버전
- **요약:** 대시보드가 마운트될 때 `GET /api/v1/references`를 호출하고, 모달 저장 시 `POST /api/v1/references`를 호출해 서버 응답으로 목록을 갱신
- **난이도:** M
- **리스크:** 낮음
- **장점:**
  - 기존 구조 변경이 작음
  - 이번 사이클의 핵심 문제를 가장 빨리 해결
  - 테스트 추가 범위가 명확함
- **단점:**
  - 인증은 여전히 mock 상태
  - 검색/필터/페이지네이션은 후속 작업 필요
  - API 호출 로직이 추후 확장 시 재정리 필요할 수 있음

### 접근법 B: 이상적 아키텍처
- **요약:** API 클라이언트 레이어, 로딩/에러 상태 공통화, 서버 인증 컨텍스트까지 함께 정리해 조회/저장/인증을 한 번에 연결
- **난이도:** L
- **리스크:** 중간
- **장점:**
  - 이후 검색/필터/인증 확장이 쉬움
  - 데이터 흐름이 더 일관됨
  - 네트워크 상태 관리 재사용 가능
- **단점:**
  - 이번 사이클 범위를 넘겨 일정이 커짐
  - 인증까지 엮이면 실패 지점이 늘어남
  - 현재 제품 단계에 비해 과설계 위험이 있음

### 추천
- **추천 접근법:** 접근법 A
- **이유:** 1인 개발 기준으로 가장 위험한 상태는 "저장된 것처럼 보이지만 실제로는 저장되지 않는 UX"입니다. 이번 사이클에서는 영속 저장 신뢰성 회복이 목적이므로, 인증과 검색을 억지로 합치지 않고 저장/조회만 먼저 닫는 것이 최선입니다.

## 7. ⚠️ 가정 (Assumptions)
- 현재 로그인 사용자는 계속 mock 사용자 `user-1234`로 간주한다.
- Empty State는 온보딩 투어가 아니라 간단한 안내 문구 + 레퍼런스 추가 CTA로 충분하다.
- 자동 재시도는 넣지 않고, 사용자가 `다시 시도`를 명시적으로 누르는 방식만 허용한다.
- 저장 멱등성 키나 중복 링크 방지는 이번 사이클에 넣지 않는다. 대신 프론트의 중복 제출 차단만 구현한다.
- API 응답 스키마는 기존 핸들러 구조를 최대한 유지한다.
