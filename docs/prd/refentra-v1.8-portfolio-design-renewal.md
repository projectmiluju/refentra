# [PRD] 포트폴리오용 랜딩·대시보드 디자인 리뉴얼

**Status:** Approved
**Date:** 2026-03-27
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** 현재 Refentra는 기능적으로는 MVP 흐름을 갖췄지만, 외부 포트폴리오 채널에서 “이 사람에게 웹사이트 제작을 맡기고 싶다”는 인상으로 연결될 만큼 시각적 설득력이 강하지는 않다. 이번 작업의 목적은 서비스 설명 중심 UI에서 벗어나, 브랜딩 중심 랜딩 페이지와 완성도 높은 제품형 대시보드를 통해 디자인 감각과 반응형 구현 역량을 동시에 보여주는 것이다.
- **타겟 워크플로우:** 방문자는 랜딩 첫 화면에서 강한 브랜드 인상을 받는다. 이후 문제 제기, 기능 하이라이트, 대시보드 미리보기, CTA 흐름을 따라가며 제품 완성도를 인지한다. 포트폴리오 모드에서는 샘플 데이터 기반 대시보드와 레퍼런스 추가 모달 상호작용을 통해 “정적인 시안”이 아닌 “실제로 제작 가능한 웹앱”이라는 인상을 받는다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] 랜딩 페이지를 브랜딩 중심 포트폴리오용 구조로 전면 리뉴얼해야 한다.
- [ ] 랜딩 섹션 순서는 최소 `히어로 → 문제 제기 → 기능 하이라이트 → 대시보드 미리보기 → CTA`를 포함해야 한다.
- [ ] 대시보드는 데스크톱과 모바일 모두에서 포트폴리오용 완성형 화면으로 보이도록 반응형 리뉴얼해야 한다.
- [ ] 포트폴리오 모드에서는 샘플 레퍼런스 데이터가 기본 노출되어야 하며, 빈 상태를 기본 시연 경로로 사용하지 않는다.
- [ ] 샘플 데이터는 정적인 스크린샷용 데이터가 아니라 `검색`, `태그 필터`, `페이지네이션`, `카드 리스트 변화`에 반응하는 인터랙티브 데이터여야 한다.
- [ ] 레퍼런스 추가 모달은 포트폴리오 시연 흐름에 포함되어야 한다.
- [ ] 저장 버튼 중복 클릭 시 클라이언트와 서버 정책 모두에서 실제 제품형 중복 제출 방지 정책을 유지해야 한다.
- [ ] 포트폴리오 모드에서 비로그인 사용자가 대시보드 URL에 직접 접근하면 로그인 페이지가 아니라 랜딩으로 되돌려야 한다.
- [ ] 포트폴리오 모드에서는 에러/빈 상태를 기본 시연 경로에서 제거하고, 실제 제품 모드에서만 기존 로딩/에러/빈 상태 정책을 유지해야 한다.
- [ ] 기존 실제 제품 모드의 검색/필터/페이지네이션/인증 계약은 깨지지 않아야 한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**
  - 랜딩: `Initial → Hero Viewed → Section Scroll → CTA`
  - 포트폴리오 대시보드: `Sample Ready → Search/Filter Applied → Page Changed → Modal Opened → Saving → Saved`
  - 접근 정책: `Guest + Portfolio Dashboard URL → Redirect Landing`
  - 제품 모드: `Idle → Loading → Success | Empty | Error`
- **핵심 데이터 필드:**
  - 모드 구분: `portfolio mode` / `product mode`
  - 샘플 레퍼런스: `id`, `title`, `description`, `url`, `tags`, `created_at`, `thumbnail/visual tone`
  - 대시보드 조회 상태: `search`, `tags[]`, `page`, `limit`
  - 모달 제출 상태: `idle`, `submitting`, `submitted`, `error`
  - 제출 보호 키: `client submit lock`, 필요 시 `server-side idempotency guard`
- **API 엔드포인트 (예상):**
  - `GET /api/v1/references`
    - 제품 모드의 기존 검색/필터/페이지네이션 계약 유지
  - `POST /api/v1/references`
    - 제품 모드 저장 계약 유지, 중복 제출 방지 정책 명시
  - 포트폴리오 모드 데이터 제공 방식
    - 옵션 A: 프론트엔드 샘플 데이터 레이어
    - 옵션 B: 기존 API 위에 시드 데이터 노출 분기
    - 이번 PRD에서는 구현 시점 결정 대상으로 남기되, 사용자 경험은 동일해야 함

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| 포트폴리오 모드 첫 진입 시 데이터 0건 | 빈 상태 대신 샘플 데이터가 기본 노출된다 | P0 |
| 포트폴리오 모드에서 비로그인 사용자가 `/dashboard` 직접 접근 | 로그인 페이지가 아니라 랜딩으로 리다이렉트한다 | P0 |
| 포트폴리오 모드에서 서버 오류 또는 API 실패 | 기본 시연 경로에서는 노출하지 않고 완성형 화면 경험을 유지한다 | P1 |
| 실제 제품 모드에서 서버 500 또는 네트워크 실패 | 기존 에러 상태와 재시도 정책을 유지한다 | P0 |
| 사용자가 저장 버튼을 연속 클릭 | 클라이언트 버튼 잠금 + 서버 기준 중복 제출 방지 정책으로 1회만 처리한다 | P0 |
| 샘플 데이터가 많아짐 | 전체 노출 대신 기존 페이지네이션 구조를 유지한다 | P0 |
| 모바일 화면에서 카드/모달 레이아웃이 밀림 | 반응형 레이아웃 우선순위를 높여 별도 모바일 배치 규칙을 둔다 | P0 |
| 포트폴리오 모드와 실제 제품 모드 UI 정책 충돌 | 포트폴리오 모드는 완성형 시연 우선, 실제 제품 모드는 운영 안정성 우선으로 분기한다 | P1 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** 접근법 A
- **핵심 기술:**
  - 프론트엔드: React 18 + Vite + TypeScript + Tailwind CSS
    - 근거: 현재 UI 스택과 완전 호환되며, 시각 리뉴얼과 반응형 구현에 가장 낮은 전환 비용으로 대응 가능하다.
  - 백엔드: Go + Echo
    - 근거: 기존 인증/레퍼런스 API를 유지하면서 포트폴리오 모드 접근 정책과 실제 제품형 제출 정책을 공존시키기 쉽다.
  - DB: PostgreSQL
    - 근거: 기존 레퍼런스 목록, 검색, 페이지네이션 모델을 그대로 재사용할 수 있다.
  - 테스트: Vitest + Playwright
    - 근거: 반응형 UI와 핵심 사용자 흐름 회귀를 함께 검증하는 현재 도구 체계와 일치한다.
  - ⚠️ 기존 프로젝트 스택과의 호환성: `호환`

### 접근법 A: 최소 생존 버전
- **요약:** 기존 앱 구조를 유지한 채 포트폴리오 모드를 추가하고, 랜딩/대시보드/모달을 포트폴리오용 시각 경험으로 재설계한다.
- **난이도:** M
- **리스크:** 낮음
- **장점:**
  - 현재 코드와 문서 흐름을 그대로 활용할 수 있다.
  - 빠르게 포트폴리오용 결과물을 만들 수 있다.
  - 기존 인증/검색/페이지네이션 계약을 재사용할 수 있다.
- **단점:**
  - 포트폴리오 모드와 실제 제품 모드 분기 관리가 필요하다.
  - 기존 대시보드 구조 제약을 일부 그대로 안고 간다.
  - 시각 자유도가 완전 신규 프레젠테이션 계층보다 낮다.

### 접근법 B: 이상적 아키텍처
- **요약:** 포트폴리오용 랜딩과 데모 대시보드를 제품 앱과 느슨하게 분리된 프레젠테이션 계층으로 재설계한다.
- **난이도:** L
- **리스크:** 중간
- **장점:**
  - 브랜드 인상을 극대화하기 쉽다.
  - 제품 모드와 포트폴리오 모드 역할을 더 명확히 분리할 수 있다.
  - 향후 포트폴리오 템플릿 자산으로 재사용하기 쉽다.
- **단점:**
  - 1인 개발 기준으로 구현 비용이 커진다.
  - 기존 제품 화면과 표현 계층 중복이 생길 수 있다.
  - 이번 사이클에서 과설계로 흐를 가능성이 높다.

### 추천
- **접근법 A**를 추천한다.
- 이유: 지금 필요한 것은 “의뢰로 이어질 만큼 인상적인 결과물을 빠르게 만드는 것”이다. 포트폴리오 전용 앱을 새로 분리하기보다, 기존 제품 위에 강한 브랜드 경험과 시연 흐름을 얹는 편이 1인 개발 기준으로 비용 대비 효과가 가장 높다.

## 6. 아웃 오브 스코프 (Out of Scope)
| 제외 기능 | 제외 이유 | 예상 도입 시기 |
|----------|----------|-------------|
| 실제 공개 포트폴리오 배포/도메인/SEO 최적화 | 이번 목표는 캡처·시연용 리뉴얼이지 공개 마케팅 사이트 운영이 아니다 | 후속 배포 사이클 |
| 포트폴리오 모드 전용 별도 백엔드/DB 분리 | 현재 단계에서는 과설계다 | 후속 아키텍처 검토 시 |
| nightly full E2E 또는 actionlint 추가 | 가치가 있지만 이번 리뉴얼의 핵심 산출물은 아니다 | CI 후속 하드닝 |
| 실제 사용자 계정 체계 전환 | 이번 포트폴리오 리뉴얼 목표와 직접 연결되지 않는다 | 인증 후속 |
| 삭제/복구 플로우 설계 | 이번 사이클은 등록·탐색·시연이 핵심이며 삭제 UX는 우선순위가 낮다 | 레퍼런스 관리 후속 |

## 7. 다음 액션 플랜 (Next Actions)
- 랜딩 섹션별 정보 구조와 시각 톤을 먼저 정의한다.
- 포트폴리오 모드 진입 조건과 실제 제품 모드와의 정책 차이를 화면 단위로 정리한다.
- 대시보드 샘플 데이터 구조와 인터랙션 범위(`검색`, `태그`, `페이지네이션`, `모달 저장`)를 확정한다.
- 비로그인 사용자의 포트폴리오 대시보드 직접 접근 시 랜딩 리다이렉트 정책을 구현 범위로 분리한다.
- 중복 제출 방지 정책을 클라이언트와 서버 레이어로 나눠 명세한다.
- PRD 승인 후 이 작업을 `랜딩`, `대시보드`, `모달/저장 정책`, `테스트` 단위 GitHub Issue로 분해한다.

## 8. 가정 (Assumptions)
- ⚠️ 포트폴리오 제출은 실제 공개 URL보다 캡처/시연 중심이라고 가정한다.
- ⚠️ 포트폴리오 방문자는 서비스 설명 이해보다 디자인 완성도와 반응형 구현 인상을 더 중요하게 본다고 가정한다.
- ⚠️ 포트폴리오 모드와 실제 제품 모드는 한 코드베이스 안에서 공존할 수 있다고 가정한다.
- ⚠️ 이번 사이클에서는 삭제/복구 플로우보다 랜딩, 대시보드, 모달 상호작용의 시각적 완성도가 더 중요하다고 가정한다.
