# [PRD] Refentra v1.2 로컬 개발용 Docker PostgreSQL 표준화

**Status:** Approved
**Date:** 2026-03-26
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** 현재 Refentra는 레퍼런스 저장/조회 기능이 구현되어 있지만, 로컬 PostgreSQL이 이미 설치되어 있다고 가정하는 구조라 실행 환경이 없는 개발자는 즉시 막힙니다. 이번 사이클의 목표는 Docker 기반 PostgreSQL 개발 환경을 표준화해 누구나 동일한 방식으로 DB를 띄우고, 데모 데이터가 포함된 상태에서 앱을 바로 실행·검증할 수 있게 만드는 것입니다.
- **타겟 워크플로우:** 개발자는 저장소를 클론한 뒤 Docker로 PostgreSQL 컨테이너를 띄우고, 로컬에서 Go/Vite 앱을 실행합니다. 첫 컨테이너 생성 시 초기 스키마와 mock 사용자/레퍼런스 데모 데이터가 1회 주입되어, 앱 접속 즉시 로그인 및 저장/조회 흐름을 검증할 수 있어야 합니다. DB 컨테이너가 꺼져 있거나 초기화에 실패하면, 앱은 정상 화면 대신 설정 안내 페이지를 보여줘야 합니다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] 프로젝트는 Docker 기반 PostgreSQL 로컬 개발 환경을 공식 지원해야 한다.
- [ ] DB 실행은 `docker-compose` 또는 동등한 Docker Compose 방식으로 표준화해야 한다.
- [ ] 앱(Go 서버, Vite 프론트)은 컨테이너가 아니라 로컬 프로세스로 실행해야 한다.
- [ ] `.env.example` 또는 동등한 환경변수 문서에 DB 접속 정보가 명시되어야 한다.
- [ ] PostgreSQL 컨테이너 첫 생성 시 초기 스키마와 데모 데이터가 1회만 주입되어야 한다.
- [ ] 데모 데이터에는 현재 mock 로그인 사용자와 예시 레퍼런스 3~5개가 포함되어야 한다.
- [ ] 개발자가 데모 데이터를 다시 초기화하고 싶다면 Docker volume 삭제 후 전체 재생성 절차로 되돌릴 수 있어야 한다.
- [ ] DB 컨테이너가 꺼져 있거나 초기화 실패 시, 앱은 대시보드나 빈 상태를 보여주지 않고 설정 안내 페이지를 보여줘야 한다.
- [ ] 설정 안내 페이지에는 최소한 “DB가 준비되지 않았다”는 사실, 실행해야 할 명령, 확인 포인트가 포함되어야 한다.
- [ ] README는 Docker DB 실행, 앱 실행, 재초기화 절차를 복붙 가능한 명령 기준으로 설명해야 한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**
  - 앱 시작 시 DB 상태 확인: `Unknown -> Ready | Unavailable`
  - DB 미연결 시 화면 상태: `Unavailable -> SetupGuide`
  - DB 연결 후 정상 화면 진입: `Ready -> Login | Dashboard`
- **핵심 데이터 필드:**
  - 환경변수
    - `DB_HOST`
    - `DB_PORT`
    - `DB_USER`
    - `DB_PASSWORD`
    - `DB_NAME`
    - `DB_SSLMODE`
    - `DB_TIMEZONE`
  - 데모 사용자
    - `id`: mock 식별자
    - `name`
    - `email`
  - 데모 레퍼런스
    - `id`
    - `url`
    - `title`
    - `description`
    - `tags`
    - `uploader_id`
    - `created_at`
- **인프라/실행 단위:**
  - Docker 서비스
    - `postgres` 1개
  - 로컬 실행 프로세스
    - Go API 서버
    - Vite 개발 서버 또는 빌드된 프론트엔드
- **API 엔드포인트 (예상):**
  - `GET /api/v1/references`
    - DB 미연결 또는 초기화 실패 시 `503 Service Unavailable`
  - `POST /api/v1/references`
    - DB 미연결 또는 초기화 실패 시 `503 Service Unavailable`
  - `GET /api/v1/auth/me`
    - 기존 mock 사용자 반환 유지
  - `GET /api/v1/health` 또는 동등한 상태 확인 메커니즘
    - 앱이 DB 준비 여부를 명시적으로 판단할 수 있어야 함

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| Docker가 설치되지 않음 | README와 설정 안내 페이지에 Docker 설치 필요 메시지 표시 | P0 |
| DB 컨테이너가 실행 중이 아님 | 앱 시작 시 설정 안내 페이지 표시, 실행 명령 제시 | P0 |
| DB 접속 정보가 `.env`와 불일치 | 앱은 설정 안내 페이지에서 환경변수 확인을 요구 | P0 |
| PostgreSQL은 떴지만 초기화 실패 | 앱은 설정 안내 페이지에서 초기화 실패를 명시하고 로그 확인 가이드를 제공 | P0 |
| 첫 컨테이너 생성 후 데모 데이터가 없음 | 초기화 실패로 간주하고 설정 안내 페이지 또는 서버 에러 로그로 노출 | P0 |
| DB 조회/저장 API 호출 시 연결 불가 | `503 Service Unavailable`와 정형화된 에러 메시지 반환 | P0 |
| 앱이 DB 미연결 상태를 빈 목록처럼 표시 | 금지. 반드시 설정 안내 페이지로 차단 | P0 |
| 컨테이너 재시작만 했는데 데모 데이터가 중복 삽입됨 | 금지. 시드는 첫 생성 시 1회만 주입 | P0 |
| 개발자가 데모 데이터를 초기화하고 싶음 | Docker volume 삭제 후 `compose up` 재실행 절차를 공식 절차로 제공 | P1 |
| 사용자가 실수로 기존 volume을 삭제함 | 데이터가 초기화되는 것이 정상 동작임을 문서에 명시 | P1 |
| 향후 레퍼런스가 1만 건 이상으로 증가 | 이번 사이클 범위 밖. 검색/페이지네이션 PRD에서 대응 | P2 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** DB만 Docker로 표준화하고 앱은 로컬 실행
- **핵심 기술:**
  - **DB 런타임:** PostgreSQL Docker 이미지
    - 근거: 로컬 설치 의존성을 제거하고 재현 가능한 개발 환경을 가장 빠르게 만들 수 있음
  - **오케스트레이션:** Docker Compose
    - 근거: 단일 DB 서비스, volume, init script를 선언적으로 관리하기 가장 단순함
  - **앱 런타임:** Go 로컬 실행 + Vite 로컬 실행
    - 근거: 기존 개발 속도를 유지하면서 DB 재현성만 확보 가능
  - **설정 방식:** `.env` / `.env.example`
    - 근거: 현재 하드코딩 DSN을 제거하고 환경별 설정을 분리해야 함
  - **초기화 방식:** PostgreSQL init script 또는 동등한 최초 기동 스크립트
    - 근거: 첫 생성 시 1회 시드 주입 요구를 자연스럽게 만족
- **기존 프로젝트 스택과의 호환성:** 호환

### 접근법 A: 최소 생존 버전
- **요약:** Docker Compose로 PostgreSQL만 띄우고, `.env`로 로컬 Go 앱이 여기에 연결되게 만든다. 최초 기동 시 init script로 스키마와 데모 데이터를 넣고, 앱은 DB 미연결 시 설정 안내 페이지를 보여준다.
- **난이도:** M
- **리스크:** 낮음
- **장점:**
  - 현재 구조 변경이 가장 작음
  - 로컬 설치 의존성을 제거함
  - 재현성, README, QA 절차를 한 번에 개선
- **단점:**
  - Go/Vite 프로세스는 여전히 따로 실행해야 함
  - Docker 미설치 사용자는 여전히 선행 설치가 필요함
  - 앱 헬스체크와 안내 페이지 구현이 추가로 필요함

### 접근법 B: 이상적 개발 환경
- **요약:** PostgreSQL뿐 아니라 Go 앱 헬스체크, 시드 관리 커맨드, 개발용 부트스트랩 스크립트까지 포함한 통합 개발 환경을 구성한다.
- **난이도:** L
- **리스크:** 중간
- **장점:**
  - 온보딩 경험이 가장 좋음
  - 재초기화, 진단, 헬스체크 흐름이 더 명확해짐
  - 이후 CI나 E2E 환경으로 확장하기 쉬움
- **단점:**
  - 지금 필요한 문제보다 범위가 큼
  - 개발 환경 코드와 문서가 크게 늘어남
  - 1인 개발 기준 초기 비용이 더 큼

### 추천
- **추천 접근법:** 접근법 A
- **이유:** 현재 가장 큰 문제는 기능이 아니라 실행 환경 부재입니다. 1인 개발 기준으로는 DB 재현성만 먼저 해결해도 저장/조회 기능 검증이 가능해집니다. 앱 전체 컨테이너화보다 `DB만 Docker`가 속도와 유지보수성의 균형이 가장 좋습니다.
