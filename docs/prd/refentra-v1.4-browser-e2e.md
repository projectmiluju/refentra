# [PRD] 브라우저 E2E 자동화

**Status:** Approved
**Date:** 2026-03-27
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** `v0.5.0`까지 로그인, 실제 세션 인증, 레퍼런스 저장/조회, 로그아웃이 구현됐지만 브라우저 기준 회귀 검증이 없습니다. 현재는 `curl`과 단위 테스트로 핵심 플로우를 확인하고 있어, 실제 사용자 환경에서 리다이렉트, 쿠키 세션, 새로고침 후 재조회가 깨져도 늦게 발견될 수 있습니다.
- **타겟 워크플로우:** 브라우저에서 `비로그인 보호 URL 접근 차단 → 로그인 → 대시보드 진입 → 온보딩 또는 기존 데이터 확인 → 레퍼런스 저장 → 새로고침 후 재조회 확인 → 로그아웃 → 재접근 차단`까지 자동 검증합니다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] Playwright 기반 브라우저 E2E 테스트 환경을 추가한다.
- [ ] 비로그인 사용자가 `/dashboard`에 직접 접근하면 로그인 페이지로 리다이렉트되는 시나리오를 검증한다.
- [ ] mock 사용자 로그인 성공 후 대시보드에 진입하는 시나리오를 검증한다.
- [ ] 로그인 직후 레퍼런스가 0건이면 온보딩 가이드가 보이는 시나리오를 검증한다.
- [ ] 레퍼런스 저장 성공 후 목록 반영과 브라우저 새로고침 뒤 재조회 유지 여부를 검증한다.
- [ ] 저장 버튼 중복 클릭 시 프론트가 두 번째 제출을 막고 결과적으로 1건만 생성되는 시나리오를 검증한다.
- [ ] 로그아웃 후 로그인 페이지로 이동하고, 뒤로가기 또는 `/dashboard` 재접근 시 다시 로그인 페이지로 리다이렉트되는 시나리오를 검증한다.
- [ ] 실패 시 자동 재시도 없이 에러 상태와 `다시 시도` 동작을 검증할 수 있는 구조를 마련한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**
  - 앱 접근: `Unauthenticated -> RedirectToLogin`
  - 로그인: `Idle -> Submitting -> Authenticated | Error`
  - 대시보드 조회: `Loading -> Onboarding | Success | Error`
  - 레퍼런스 저장: `Idle -> Saving -> Success | Error`
  - 로그아웃: `Authenticated -> LoggingOut -> Unauthenticated`
- **핵심 데이터 필드:**
  - 로그인 입력: `email`, `password`
  - 생성 데이터: `title`, `url`, `summary`, `tags[]`
  - E2E 테스트 식별자: 각 테스트 실행 시각 또는 랜덤 접미사를 포함한 고유 `title`, `url`
  - 검증 상태: 현재 URL, 표시 메시지, 저장된 레퍼런스 텍스트, 로그아웃 후 접근 차단 여부
- **API 엔드포인트 (예상):**
  - `GET /api/v1/health`
  - `POST /api/v1/auth/login`
  - `GET /api/v1/auth/me`
  - `POST /api/v1/auth/logout`
  - `GET /api/v1/references`
  - `POST /api/v1/references`
- **테스트 데이터 정책:**
  - PostgreSQL/Redis 컨테이너와 기존 데모 데이터를 유지한 채 실행한다.
  - 테스트마다 고유 제목/URL을 생성해 기존 데이터와 충돌을 피한다.
  - DB volume 리셋은 이번 사이클 범위에 포함하지 않는다.

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| 비로그인 사용자가 `/dashboard` 직접 접근 | 로그인 페이지로 리다이렉트하고 로그인 폼이 보여야 한다 | P0 |
| 로그인 실패 | 에러 메시지를 보여주고 입력값은 유지한다 | P0 |
| 레퍼런스 0건 | 빈 목록 대신 온보딩 가이드를 표시한다 | P0 |
| 저장 중 버튼 중복 클릭 | 첫 요청만 유효, 버튼 비활성화 상태를 확인한다 | P0 |
| 저장 후 새로고침 | 방금 생성한 고유 제목/URL이 다시 보여야 한다 | P0 |
| 서버 `500` 또는 `503` | 자동 재시도 없이 에러 메시지와 `다시 시도` 버튼을 표시한다 | P1 |
| 로그아웃 후 브라우저 뒤로가기 | 보호 화면 복귀 대신 로그인 페이지로 다시 밀려나야 한다 | P0 |
| 세션 만료 | 로그인 페이지 이동과 `세션이 만료되었습니다` 메시지를 추후 별도 시나리오로 확장 가능하게 구조화한다 | P1 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** 접근법 A, Playwright 기반 최소 생존 버전
- **핵심 기술:**
  - **프론트엔드:** 기존 React/Vite 유지
    - 근거: 현재 사용자 플로우가 모두 이 앱 안에 있어 브라우저 자동화 대상이 명확합니다.
  - **백엔드:** 기존 Go/Echo 유지
    - 근거: 로그인, 보호 API, 저장/조회 흐름이 이미 구현돼 있어 브라우저 E2E는 기존 서버 위에서 검증하면 됩니다.
  - **DB/세션 저장소:** 기존 Docker Compose PostgreSQL + Redis 유지
    - 근거: 실제 세션과 저장/조회가 컨테이너 상태에 의존하므로 E2E도 동일 환경을 기준으로 해야 합니다.
  - **E2E 도구:** Playwright
    - 근거: 리다이렉트, 브라우저 새로고침, 쿠키 세션, 네트워크 안정성 확인에 가장 적합합니다.
  - **기존 프로젝트 스택과의 호환성:** 호환

## 6. 아웃 오브 스코프 (Out of Scope)
| 제외 기능 | 제외 이유 | 예상 도입 시기 |
|----------|----------|-------------|
| 테스트 전용 DB 리셋/시드 자동화 | 이번 목표는 핵심 회귀선 확보이며, 인프라 재현성 고도화는 후속 단계가 더 적절함 | v1.5+ |
| CI 파이프라인에서 Playwright 자동 실행 | 로컬 회귀선 확보가 먼저이며, CI 연결은 이후 운영 단계에서 다루는 편이 효율적임 | v1.5+ |
| 검색/태그 필터/페이지네이션 E2E | 아직 기능 자체가 목업 수준이라 브라우저 E2E보다 기능 구현이 선행돼야 함 | 해당 기능 구현 이후 |
| 실제 계정/권한 체계 검증 | 현재는 mock 사용자 1명 기반 세션 검증이 우선임 | 인증 고도화 사이클 |
| 세션 만료/리이슈 전체 브라우저 시뮬레이션 | 구현 가능하지만 이번 사이클 핵심 플로우보다 우선순위가 낮음 | v1.5+ |
