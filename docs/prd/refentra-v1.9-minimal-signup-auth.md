# [PRD] Refentra v1.9 최소 회원가입 기반 실제 사용자 계정 도입

**Status:** Approved
**Date:** 2026-03-29
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** 현재 Refentra는 포트폴리오 제출용 랜딩, 대시보드, 모달, 제품/포트폴리오 모드 분리까지는 정리됐지만, 실제 사용자 계정 체계는 여전히 mock 사용자 1명에 기대고 있다. 이 상태에서는 “회원가입 가능한 실제 제품”으로 보이기 어렵고, 사용자별 데이터 소유권도 명확하지 않다. 이번 사이클의 목표는 이메일 인증이나 계정 관리 전체를 한 번에 완성하는 것이 아니라, 최소 정보 기반 회원가입과 로그인 흐름을 실제 사용자 계정 기준으로 전환해 제품 MVP의 신뢰선을 끌어올리는 것이다.
- **타겟 워크플로우:** 비로그인 사용자는 랜딩에서 제품을 살펴보다가 회원가입 또는 로그인으로 진입한다. 회원가입 폼에서 `name`, `email`, `password`를 입력하면 계정이 생성되고 즉시 로그인된다. 로그인된 사용자는 빈 상태 대시보드에서 첫 레퍼런스를 추가할 수 있고, 이후 조회/저장 흐름은 기존 제품 모드 계약을 그대로 사용한다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] 회원가입은 최소 정보 `name`, `email`, `password`만 받아야 한다.
- [ ] 비밀번호 정책은 이번 사이클에서 `8자 이상`만 강제한다.
- [ ] 이메일 인증, 본인 확인, 비밀번호 재설정은 이번 사이클에 포함하지 않는다.
- [ ] 회원가입 성공 직후 사용자는 자동 로그인되어야 한다.
- [ ] 로그인은 더 이상 mock 사용자 1명 고정이 아니라 실제 저장된 사용자 계정 기준으로 동작해야 한다.
- [ ] 비로그인 사용자가 `/dashboard` 같은 보호 URL에 직접 접근하면 로그인 페이지가 아니라 랜딩(`/`)으로 리다이렉트해야 한다.
- [ ] 로그인한 사용자가 레퍼런스 0건 상태로 처음 진입하면 빈 상태 대시보드를 봐야 한다.
- [ ] 비로그인 사용자는 계속 랜딩/온보딩 중심 흐름으로 남아야 한다.
- [ ] 같은 이메일로 중복 회원가입을 시도하면 클라이언트와 서버 모두에서 중복 제출을 막아야 한다.
- [ ] 회원가입 버튼은 제출 직후 잠겨야 하며, 같은 요청을 연속으로 보내지 않아야 한다.
- [ ] 서버는 이메일 unique 제약을 보장해야 하고, 중복 가입 시 명시적 오류를 반환해야 한다.
- [ ] 회원가입/로그인 이후 세션 유지, 보호 라우트, 레퍼런스 저장/조회 계약은 기존 실제 제품 모드와 호환되어야 한다.
- [ ] 기존 포트폴리오 모드(`/dashboard?mode=portfolio`)는 이번 사용자 계정 도입 후에도 계속 동작해야 한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**
  - 회원가입: `Idle -> Submitting -> Authenticated | DuplicateEmail | Error`
  - 로그인: `Idle -> Submitting -> Authenticated | Error`
  - 보호 라우트 접근: `Guest + Protected Route -> Redirect Landing`
  - 첫 사용자 제품 모드: `Authenticated + EmptyReferences -> Empty Dashboard`
  - 포트폴리오 모드: `Guest|Authenticated -> Portfolio Demo`
- **핵심 데이터 필드:**
  - 사용자 계정
    - `id`
    - `email`
    - `name`
    - `password_hash`
    - `created_at`
    - `updated_at`
    - `deleted_at`는 이번 사이클에 없음
  - 인증 입력값
    - `name`
    - `email`
    - `password`
  - 프론트 상태
    - `idle`
    - `submitting`
    - `authenticated`
    - `error`
    - `duplicate_email`
  - 보호 라우트 상태
    - `checking`
    - `authenticated`
    - `unauthenticated`
- **DB 제약 (예상):**
  - `users.email` unique index
  - 레퍼런스 `uploader_id`는 새 사용자 계정 `id`와 연결
- **API 엔드포인트 (예상):**
  - `POST /api/v1/auth/signup`
    - `name`, `email`, `password`로 계정 생성
    - 성공 시 자동 로그인과 동일한 세션 발급
  - `POST /api/v1/auth/login`
    - 저장된 사용자 계정 기준 인증
  - `POST /api/v1/auth/logout`
    - 현재 세션 폐기
  - `GET /api/v1/auth/me`
    - 현재 로그인 사용자 조회
  - `GET /api/v1/references`
    - 기존 제품 모드 계약 유지
  - `POST /api/v1/references`
    - 기존 제품 모드 계약 유지, `uploader_id`는 현재 로그인 사용자 기준

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| 사용자가 비밀번호 8자 미만으로 제출 | 클라이언트에서 즉시 오류 표시, 요청 전송 차단 | P0 |
| 사용자가 잘못된 이메일 형식 제출 | 클라이언트에서 즉시 오류 표시, 요청 전송 차단 | P0 |
| 회원가입 버튼을 짧게 두 번 누름 | 첫 요청 직후 버튼 비활성화, 중복 요청 차단 | P0 |
| 이미 가입된 이메일로 회원가입 시도 | 서버에서 unique 보장, `이미 가입된 이메일입니다` 오류 표시 | P0 |
| 회원가입 API가 500을 반환 | 자동 재시도 없이 오류 메시지와 다시 시도 버튼 표시 | P0 |
| 회원가입 중 네트워크 끊김 | 자동 재시도 없이 오류 메시지와 다시 시도 버튼 표시 | P0 |
| 로그인 정보가 틀림 | 오류 메시지 표시, 입력값 유지 | P0 |
| 비로그인 사용자가 `/dashboard` 직접 접근 | 랜딩(`/`)으로 리다이렉트 | P0 |
| 로그인 직후 레퍼런스가 0건 | 온보딩이 아니라 빈 상태 대시보드 표시 | P1 |
| 포트폴리오 모드와 제품 모드 URL이 섞임 | `mode=portfolio`가 있으면 데모 레이어 우선, 그 외는 제품 모드 | P1 |
| 회원가입 성공 후 세션 발급 실패 | 계정 생성 여부와 세션 실패 메시지를 명확히 분리하거나, 트랜잭션 정책상 전체 실패 처리 | P1 |
| 장기적으로 계정 삭제가 필요해짐 | 별도 계정 관리 PRD에서 soft delete 여부 포함 재설계 | P2 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** 접근법 A
- **핵심 기술:**
  - 프론트엔드: React 18 + Vite + TypeScript
    - 근거: 현재 로그인/보호 라우트 구조 위에 회원가입 화면과 상태만 추가하면 된다.
  - 백엔드: Go + Echo
    - 근거: 기존 인증 API 구조를 유지하면서 `signup`만 추가하기 쉽다.
  - DB: PostgreSQL
    - 근거: `users` 테이블과 이메일 unique 제약을 관리하기 적합하다.
  - 세션 저장소: Redis
    - 근거: 현재 세션/refresh 구조를 재사용하는 편이 가장 안전하다.
  - 테스트: Vitest + Playwright + Go test
    - 근거: 폼 검증, 보호 라우트, 브라우저 플로우, 백엔드 unique 제약을 각 레이어에서 나눠 검증할 수 있다.
  - ⚠️ 기존 프로젝트 스택과의 호환성: `호환`

### 접근법 A: 최소 생존 버전
- **요약:** 기존 인증 구조를 유지한 채 mock 사용자 1명을 실제 `users` 테이블 기반 계정으로 대체하고, 회원가입 API와 로그인 검증만 추가한다.
- **난이도:** M
- **리스크:** 낮음
- **장점:**
  - 현재 제품 구조를 크게 흔들지 않는다.
  - 회원가입 가능한 실제 제품 흐름을 빠르게 확보할 수 있다.
  - 포트폴리오 모드와 제품 모드를 그대로 공존시킬 수 있다.
- **단점:**
  - 이메일 인증, 비밀번호 재설정, 계정 관리가 비어 있다.
  - 장기적으로 auth 도메인 정리가 한 번 더 필요하다.
  - 회원가입 성공 후 세션 발급 실패 정책을 명확히 정해야 한다.

### 접근법 B: 이상적 아키텍처
- **요약:** 회원가입, 이메일 인증, 비밀번호 재설정, 계정 삭제/복구, 권한 정책까지 한 번에 설계한다.
- **난이도:** L
- **리스크:** 중간
- **장점:**
  - 실제 서비스 운영 구조에 더 가깝다.
  - 이후 계정 관리 기능 재작업이 적다.
  - 인증 도메인 경계가 더 명확해진다.
- **단점:**
  - 이번 사이클 범위를 과하게 키운다.
  - 제품 MVP보다 계정 관리가 더 큰 프로젝트가 된다.
  - 1인 개발 기준으로 포트폴리오/제품 흐름 모두 늦어진다.

### 추천
- **추천 접근법:** 접근법 A
- **이유:** 현재 가장 큰 약점은 “회원가입 가능한 실제 사용자 계정이 없다”는 점이다. 이번 사이클에서는 그 약점을 가장 작은 비용으로 닫는 것이 맞고, 이메일 인증/계정 삭제까지 넣는 것은 과설계에 가깝다.

| 비밀번호 재설정/이메일 찾기 | 인증 보조 기능까지 넣으면 범위가 과도하게 커짐 | 후속 인증 하드닝 |

## 7. 다음 액션 플랜 (Next Actions)
- 사용자 계정 테이블과 이메일 unique 정책을 먼저 정의한다.
- 회원가입 API(`POST /api/v1/auth/signup`)와 로그인 API 계약 차이를 정리한다.
- 회원가입 성공 직후 자동 로그인 정책을 세션 발급 흐름과 함께 구현 범위로 묶는다.
- 보호 라우트가 계속 랜딩으로 리다이렉트되도록 현재 포트폴리오/제품 정책과 정합성을 맞춘다.
- 로그인/회원가입 화면의 빈 상태와 에러 상태를 UI 명세로 정리한다.
- PRD 승인 후 이 작업을 `DB/인증 API`, `회원가입/로그인 UI`, `테스트` 단위 GitHub Issue로 분해한다.

## 8. 가정 (Assumptions)
- ⚠️ 이번 사이클은 실제 공개 서비스가 아니라 로컬/데모 제품 품질을 높이는 단계라고 가정한다.
- ⚠️ 이메일 인증이 없어도 현재 단계에서는 충분하다고 가정한다.
- ⚠️ 가입 직후 자동 로그인 UX가 별도 검증 단계보다 더 중요하다고 가정한다.
- ⚠️ 로그인 사용자의 첫 진입은 온보딩보다 빈 상태 대시보드가 더 제품답다고 가정한다.
