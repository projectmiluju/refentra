# [PRD] Refentra v2.0 레퍼런스 삭제/복구 라이프사이클

**Status:** Approved
**Date:** 2026-04-04
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** 현재 Refentra는 레퍼런스 저장, 조회, 검색, 태그 필터, 페이지네이션, 최소 회원가입/로그인까지 구현되어 있지만, 한 번 저장한 레퍼런스를 정리하는 공식 방법이 없습니다. 이 상태에서는 잘못 저장한 링크, 더 이상 유효하지 않은 자료, 중복 등록된 항목을 제품 안에서 정리할 수 없어 아카이브 신뢰도가 떨어집니다. 이번 사이클의 목표는 레퍼런스를 "실수로 지워도 복구 가능하게" 정리할 수 있는 최소 삭제/복구 정책을 도입하는 것입니다.
- **타겟 워크플로우:** 로그인한 사용자는 자신이 저장한 레퍼런스 카드에서 삭제 액션을 실행합니다. 시스템은 즉시 하드 삭제하지 않고 soft delete 처리하여 일반 목록에서는 숨기고, 사용자는 제한된 기간 안에 복구할 수 있습니다. 복구 가능 기간이 지나면 운영 정리 작업 또는 후속 정책에 따라 영구 삭제 대상으로 전환됩니다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] 로그인한 사용자는 자신이 업로드한 레퍼런스만 삭제할 수 있어야 한다.
- [ ] 삭제는 기본적으로 hard delete가 아니라 soft delete로 처리해야 한다.
- [ ] soft delete된 레퍼런스는 기본 대시보드 목록, 검색 결과, 태그 집계에서 즉시 제외되어야 한다.
- [ ] 삭제 직후 사용자는 복구 가능 여부와 남은 정책을 이해할 수 있는 피드백을 받아야 한다.
- [ ] 사용자는 삭제한 레퍼런스를 삭제 시각 기준 `24시간` 안에 복구할 수 있어야 한다.
- [ ] 복구는 삭제한 본인만 수행할 수 있어야 한다.
- [ ] 비로그인 사용자가 삭제/복구 API를 직접 호출하면 `401 Unauthorized`를 반환해야 한다.
- [ ] 다른 사용자의 레퍼런스를 삭제/복구하려 하면 존재 유무를 과하게 노출하지 말고 거부해야 한다.
- [ ] 삭제 버튼 연속 클릭, 복구 버튼 연속 클릭 시 중복 처리되지 않도록 클라이언트와 서버 모두에서 방어해야 한다.
- [ ] 페이지네이션/검색/태그 필터와 함께 동작할 때도 삭제/복구 후 목록 수, 현재 페이지, 태그 집계가 일관되게 갱신되어야 한다.
- [ ] 포트폴리오 모드의 데모 데이터는 실제 제품 데이터 라이프사이클 정책과 분리해야 한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**
  - 일반 목록: `Loading -> Success | Empty | Error`
  - 삭제: `Idle -> Confirming -> Deleting -> Deleted | Error`
  - 복구: `Idle -> Restoring -> Restored | Error`
  - 복구 유예 만료: `Deleted -> Expired`
- **핵심 데이터 필드:**
  - `references.deleted_at`
    - soft delete 시각
  - `references.deleted_by`
    - 삭제 수행 사용자 ID
  - `references.restore_until`
    - 복구 허용 만료 시각
    - 기본값: 삭제 시각 기준 `24시간`
  - `references.updated_at`
    - 삭제/복구 시 갱신
  - 기존 유지 필드
    - `id`
    - `url`
    - `title`
    - `description`
    - `tags`
    - `uploader_id`
    - `created_at`
- **프론트엔드 상태 모델:**
  - 목록 아이템 액션 상태
    - `idle`
    - `confirming_delete`
    - `deleting`
    - `deleted`
    - `restoring`
    - `error`
  - 삭제 피드백 상태
    - `toast_visible`
    - `undo_available`
    - `undo_expired`
- **API 엔드포인트 (예상):**
  - `DELETE /api/v1/references/:id`
    - 성공: `204 No Content`
    - 실패: `401 Unauthorized`, `403 Forbidden` 또는 `404 Not Found`, `409 Conflict`, `500 Internal Server Error`
  - `POST /api/v1/references/:id/restore`
    - 성공: `200 OK`
    - 응답: 복구된 레퍼런스 객체
    - 실패: `401 Unauthorized`, `403 Forbidden` 또는 `404 Not Found`, `410 Gone`, `409 Conflict`, `500 Internal Server Error`
  - `GET /api/v1/references`
    - 기본값은 active 레퍼런스만 반환
    - 후속 확장 시 `include_deleted=mine` 같은 휴지통 조회 파라미터를 고려할 수 있으나 이번 사이클 기본 요구사항은 아님

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| 사용자가 삭제 버튼을 짧게 두 번 누름 | 첫 요청 직후 버튼 잠금, 서버는 이미 삭제된 상태를 중복 성공 또는 `409` 없이 멱등적으로 처리 | P0 |
| 삭제 후 현재 페이지에 더 이상 아이템이 없음 | 현재 페이지가 비면 직전 유효 페이지로 이동하거나 `1페이지`로 보정 | P0 |
| 삭제 후 검색/태그 결과가 0건이 됨 | 일반 빈 상태가 아니라 현재 필터 기준 `No results` 상태를 표시 | P0 |
| 비로그인 사용자가 삭제/복구 API 접근 | `401 Unauthorized` 반환 후 랜딩 또는 로그인 진입 흐름 유지 | P0 |
| 다른 사용자의 레퍼런스를 삭제/복구 시도 | 권한 없음으로 거부. 외부 사용자가 ID 존재 여부를 추측하기 어렵게 `404` 우선 정책 권장 | P0 |
| 이미 삭제된 레퍼런스를 다시 삭제 | 서버는 멱등 처리. UI는 중복 에러를 과도하게 노출하지 않음 | P1 |
| 복구 가능 기간이 지난 항목 복구 시도 | `410 Gone` 또는 동등한 명시적 오류와 함께 "복구 기간이 만료되었습니다" 메시지 표시 | P0 |
| 복구 직후 현재 검색/태그 조건과 맞지 않음 | 복구는 성공 처리하되 현재 목록에 바로 나타나지 않을 수 있음을 명시 | P1 |
| 서버 500 또는 DB 오류 | 액션 영역에 실패 메시지와 재시도 제공. 기존 목록은 유지 | P0 |
| 첫 로그인 직후 데이터가 0건 | 기존 온보딩 빈 상태 유지. 삭제/복구 기능 때문에 별도 빈 상태를 추가하지 않음 | P1 |
| 포트폴리오 모드에서 삭제 액션 노출 | 실제 제품 데이터 정책과 혼동되므로 이번 사이클에서는 비노출 또는 데모 전용 로컬 처리로 분리 | P0 |
| 영구 삭제 스케줄러가 아직 없음 | restore window 만료 후 즉시 hard delete하지 않고 "영구 삭제 후보" 상태로 남겨 후속 운영 작업에서 처리 | P1 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** 접근법 A
- **핵심 기술:**
  - **프론트엔드:** React 18 + Vite + TypeScript
    - 근거: 현재 대시보드 카드 액션, 토스트, 페이지 보정 로직에 가장 적은 변경으로 붙일 수 있다.
  - **백엔드:** Go + Echo
    - 근거: 기존 인증 미들웨어와 `/api/v1/references` 계약 위에 삭제/복구 엔드포인트를 자연스럽게 추가할 수 있다.
  - **DB:** PostgreSQL + GORM soft delete
    - 근거: 현재 `references` 모델에 이미 `deleted_at`이 있고, 조회 SQL도 active 데이터만 노출하는 방향과 호환된다.
  - **세션/권한:** 기존 JWT + Redis 세션
    - 근거: 삭제/복구는 현재 로그인 사용자 기준 `uploader_id` 소유권 검증만 추가하면 된다.
  - **테스트:** Go test + Vitest + Playwright
    - 근거: 권한, 목록 갱신, 페이지 보정, 실제 브라우저 액션을 계층별로 나눠 검증할 수 있다.
- **⚠️ 기존 프로젝트 스택과의 호환성:** 호환

### 접근법 A: 최소 생존 버전
- **요약:** 레퍼런스는 soft delete만 도입하고, 삭제 시각 기준 `24시간` 복구 유예를 제공한다. 휴지통 전용 화면은 만들지 않고 삭제 직후 액션 또는 간단한 내 항목 복구 진입점만 둔다.
- **난이도:** M
- **리스크:** 낮음
- **장점:**
  - 현재 `deleted_at` 기반 모델과 가장 잘 맞는다.
  - 실수 삭제를 방지하면서도 구현 범위를 통제하기 쉽다.
  - 페이지네이션/검색/태그 계약을 크게 흔들지 않는다.
- **단점:**
  - 휴지통 목록이 없으면 오래된 삭제 항목을 다시 찾기 어렵다.
  - 운영 차원의 영구 삭제 정리 정책은 후속 작업으로 남는다.
  - 즉시 복구 UX 설계를 잘못하면 사용자가 "어디서 복구하는지" 헷갈릴 수 있다.

### 접근법 B: 이상적 아키텍처
- **요약:** soft delete + 휴지통 화면 + 복구 유예 + 만료 후 background purge까지 포함한 전체 라이프사이클을 설계한다.
- **난이도:** L
- **리스크:** 중간
- **장점:**
  - 제품 관점에서 가장 완전한 데이터 정리 경험을 제공한다.
  - 삭제/복구/만료/영구 삭제가 문서상 명확하다.
  - 향후 계정 관리 및 감사 로그와 연결하기 쉽다.
- **단점:**
  - 1인 개발 기준 현재 제품 단계에 비해 범위가 크다.
  - 휴지통 UI, 백그라운드 정리, 운영 정책까지 한 번에 필요하다.
  - 테스트 축이 늘어나 초기 구현 속도를 늦춘다.

### 접근법 C: 즉시 hard delete
- **요약:** 삭제 요청이 들어오면 DB에서 즉시 제거하고 복구 기능은 제공하지 않는다.
- **난이도:** S
- **리스크:** 높음
- **장점:**
  - 구현이 가장 단순하다.
  - 휴지통/만료 정책이 필요 없다.
- **단점:**
  - 실수 삭제 복구가 불가능하다.
  - 현재 모델의 `deleted_at` 존재와도 방향이 맞지 않는다.
  - 제품 신뢰성과 운영 복원력이 낮다.

## 7. 다음 액션 플랜 (Next Actions)
- `soft delete + 24시간 복구 유예`를 이번 사이클 기본 정책으로 확정한다.
- 삭제/복구 API 권한 정책을 `uploader_id == current_user_id` 기준으로 고정한다.
- 대시보드 카드 수준 삭제 액션, 확인 UI, 성공 후 undo 피드백 범위를 정리한다.
- 삭제 후 페이지네이션 보정과 태그 집계 갱신 규칙을 구현 명세로 고정한다.
- PRD 승인 후 이 작업을 `백엔드 삭제/복구 API`, `대시보드 삭제/복구 UI`, `테스트` 단위 GitHub Issue로 분해한다.

## 8. 가정 (Assumptions)
- ⚠️ 사용자가 가장 자주 겪는 문제는 "잘못 저장한 레퍼런스 정리"이며, 대규모 운영 감사보다 자기 데이터 관리가 더 중요하다고 가정한다.
- ⚠️ 현재 제품 단계에서는 hard delete보다 soft delete가 더 안전하고, 이미 존재하는 `deleted_at` 필드와도 맞는다고 가정한다.
- ⚠️ 복구는 삭제 시각 기준 `24시간` 안에서면 충분하고, 별도 휴지통 화면 없이도 MVP 검증이 가능하다고 가정한다.
- ⚠️ 다른 사용자의 데이터는 숨기는 편이 더 안전하므로 권한 오류는 `404` 우선 정책이 적절하다고 가정한다.
- ⚠️ 실제 배포 전 단계이므로 만료 후 영구 삭제는 즉시 자동화하지 않아도 된다고 가정한다.
