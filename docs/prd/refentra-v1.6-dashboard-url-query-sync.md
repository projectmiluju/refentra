# [PRD] 대시보드 검색/필터/페이지 URL 쿼리 동기화

**Status:** Approved
**Date:** 2026-03-27
**Branch:** main

## 1. 개요 (Overview)
- **배경 및 목적:** 현재 대시보드는 검색어, 태그 필터, 현재 페이지를 내부 상태로만 관리합니다. 그래서 새로고침, 브라우저 뒤로가기, 링크 공유 시 사용자가 보던 탐색 상태를 그대로 복원할 수 없습니다. 이번 기능의 목적은 “지금 보고 있는 레퍼런스 탐색 상태”를 URL로 표현해, 새로고침과 공유 후에도 동일한 결과를 다시 볼 수 있게 만드는 것입니다.
- **타겟 워크플로우:** 사용자가 검색어를 입력하고, 태그를 고르고, 페이지를 이동한 뒤 주소를 복사하거나 새로고침해도 같은 결과를 즉시 복원한다.

## 2. 핵심 요구사항 (Core Requirements)
- [ ] 대시보드의 `search`, `tags`, `page` 상태를 URL 쿼리와 양방향 동기화한다.
- [ ] 페이지 최초 진입 시 URL 쿼리를 읽어 서버 조회 파라미터의 초기값으로 사용한다.
- [ ] UI에서 검색어, 태그, 페이지가 바뀌면 URL도 즉시 반영된다.
- [ ] 브라우저 새로고침, 뒤로가기/앞으로가기 시 URL 기준으로 상태를 복원한다.
- [ ] 링크 공유 시 같은 `search`, `tags`, `page` 조건의 결과를 다른 세션에서도 재현할 수 있어야 한다.
- [ ] 잘못된 페이지 번호(`0`, 음수, 숫자 아님, 범위 초과)는 예측 가능하게 보정한다.
- [ ] 일반 빈 상태와 검색 결과 없음 상태는 기존 정책을 유지한다.
- [ ] 인증이 필요한 `/dashboard` 특성상, 비로그인 사용자가 쿼리 포함 URL에 접근해도 로그인 후 같은 탐색 상태로 돌아올 수 있어야 한다.  
  ⚠️ 가정: 로그인 후 원래 요청한 쿼리 문자열을 유지한 채 `/dashboard`로 복귀한다.

## 3. 데이터 및 상태 정의 (Data & State Models)
- **상태 전이도:**  
  `URL Parsed → Loading → Success / Empty / NoResults / Error`  
  `UI Change → URL Update → Loading → Success / Empty / NoResults / Error`

- **URL 쿼리 정의:**
  - `search`: 문자열, 공백 trim 후 사용
  - `tags`: 반복 쿼리 파라미터 방식  
    예: `/dashboard?search=react&tags=Frontend&tags=Design&page=2`
  - `page`: 양의 정수, 기본값 `1`

- **핵심 데이터 필드:**
  - `searchQuery: string`
  - `selectedTags: string[]`
  - `currentPage: number`
  - `lastValidPage: number | null`  
    현재 세션에서 마지막으로 성공적으로 조회된 페이지
  - `availableTags: string[]`
  - `totalPages: number`
  - `totalCount: number`

- **API 엔드포인트 (예상):**
  - 기존 `GET /api/v1/references?search=&tags=&page=&limit=` 재사용
  - 백엔드 신규 API 추가는 없음

## 4. 예외 처리 정책 (Exception Handling)
| 상황 | 대응 방안 | 우선순위 |
|------|---------|---------|
| `page`가 숫자가 아님 | URL을 `page=1`로 보정 후 재조회 | P0 |
| `page <= 0` | URL을 `page=1`로 보정 후 재조회 | P0 |
| `page`가 마지막 페이지보다 큼 | 에러 메시지 표시 후 `lastValidPage`로 이동. `lastValidPage`가 없으면 `page=1`로 이동 | P0 |
| `search` 변경 시 기존 `page > 1` | `page=1`로 리셋 | P0 |
| 태그 변경 시 기존 `page > 1` | `page=1`로 리셋 | P0 |
| URL 쿼리에 없는 태그 값이 들어옴 | 태그는 선택 상태로 유지하되, 결과가 없으면 `검색 결과 없음` 정책 적용 | P1 |
| 서버 `500`/`503` | 기존 정책 유지: 에러 메시지 + `다시 시도` 버튼, 현재 URL 쿼리는 보존 | P0 |
| 비로그인 사용자가 쿼리 포함 `/dashboard` 접근 | 로그인 페이지로 리다이렉트 후 같은 쿼리 문자열로 복귀 | P0 |

## 5. 기술 스택 (Technology Stack)
- **선택한 접근법:** 접근법 A
- **핵심 기술:**  
  - 프론트엔드: React + `react-router-dom` `useSearchParams`  
    근거: 현재 라우팅 스택과 가장 자연스럽게 맞고 추가 의존성 없이 구현 가능
  - 백엔드: 기존 Go/Echo API 재사용  
    근거: 검색/필터/페이지네이션 API가 이미 존재하므로 URL 상태만 프론트에 반영하면 됨
  - 테스트: Vitest + Playwright 확장  
    근거: 상태 파싱과 브라우저 뒤로가기/새로고침은 브라우저 레벨 검증 가치가 높음
  - 기존 프로젝트 스택과의 호환성: `호환`

### 접근법 A: 최소 생존 버전
- **요약:** 대시보드 내부 상태를 유지하되, 초기값과 변경 결과를 `useSearchParams`로 동기화한다.
- **난이도:** M
- **리스크:** 낮음
- **장점:**
  - 기존 대시보드 구조를 크게 흔들지 않음
  - 구현 범위가 명확하고 빠름
  - 현재 API 계약을 그대로 사용 가능
- **단점:**
  - URL 상태와 내부 상태가 이중으로 존재함
  - 동기화 순서 버그를 조심해야 함

### 접근법 B: 이상적 아키텍처
- **요약:** URL 쿼리를 대시보드 상태의 단일 소스로 두고, 내부 상태는 파생값으로만 사용한다.
- **난이도:** L
- **리스크:** 중간
- **장점:**
  - 뒤로가기/새로고침/공유 동작이 더 일관됨
  - 상태 출처가 명확해 디버깅이 쉬움
  - 향후 saved search, sorting, URL 공유 기능으로 확장하기 좋음
- **단점:**
  - 초기 리팩터링 비용이 큼
  - 현재 대시보드 컴포넌트 구조를 더 많이 건드려야 함

### 추천
- **접근법 A**를 추천합니다.
- 이유: 이 프로젝트는 이미 검색/태그/페이지 서버 조회 계약이 완성돼 있고, 지금 필요한 것은 아키텍처 재작성보다 “사용자가 보던 상태를 다시 여는 기능”입니다. 1인 개발자 기준으로는 작은 변경으로 가치가 큰 쪽이 맞습니다.

## 6. 아웃 오브 스코프 (Out of Scope)
| 제외 기능 | 제외 이유 | 예상 도입 시기 |
|----------|----------|-------------|
| 정렬(sort) URL 동기화 | 현재 정렬 기능 자체가 없음 | v0.7.x 이후 |
| saved search / 즐겨찾기 필터 | URL 동기화보다 범위가 큼 | v1 이상 |
| 검색/필터 URL 상태의 로컬 저장 | 공유성과 예측 가능성보다 우선순위 낮음 | 미정 |
| 백엔드 차원의 강제 페이지 자동 보정 | 현재는 프론트 UX 정책으로 먼저 해결 가능 | 필요 시 후속 |

## 7. 다음 액션 플랜 (Next Actions)
- 대시보드 상태와 URL 쿼리 간 단방향/양방향 동기화 규칙을 구현한다.
- `search`, `tags`, `page` 변경 시 URL 업데이트 및 서버 재조회 순서를 정리한다.
- 잘못된 `page` 처리 정책을 구현한다.
  - `lastValidPage`가 있으면 그 값으로 복귀
  - 없으면 `page=1`
- Vitest에 URL 파싱, 페이지 리셋, 잘못된 페이지 보정 테스트를 추가한다.
- Playwright에 새로고침/뒤로가기/공유 URL 복원 시나리오를 추가한다.

## 8. 가정 (Assumptions)
- ⚠️ `tags`는 반복 쿼리 파라미터(`tags=A&tags=B`)로 표현한다.
- ⚠️ 비로그인 상태에서 쿼리 포함 `/dashboard` 접근 시, 로그인 후 같은 쿼리 상태로 복귀한다.
- ⚠️ 사용자가 직접 `page=999` 같은 잘못된 값을 입력한 첫 진입 상황에서는 `page=1`로 보정한다.
